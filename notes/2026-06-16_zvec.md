# zvec

- Repo: `alibaba/zvec`
- URL: https://github.com/alibaba/zvec

- Repo snapshot studied: `main` @ `841b1ed3f9fa81430048087a0accaffbe1c99211`
- Why picked today: It was on GitHub daily trending when checked, but the better reason to study it is structural: this is not "vector search from a README." It is an embedded vector database with a real C++ storage/query core, SQL-ish planning, full-text search, optional DiskANN, Python bindings, and SDK-facing extension points that make it worth stealing ideas from.

## Executive summary
zvec is an in-process vector database designed to live inside the application rather than behind a network service. The repo shape shows a serious split between runtime layers: C++ algorithms under [`src/core/algorithm/`](https://github.com/alibaba/zvec/tree/main/src/core/algorithm), collection/query/storage code under [`src/db/`](https://github.com/alibaba/zvec/tree/main/src/db), Python bindings under [`src/binding/python/`](https://github.com/alibaba/zvec/tree/main/src/binding/python), and the higher-level Python facade under [`python/zvec/`](https://github.com/alibaba/zvec/tree/main/python/zvec).

The strongest architectural idea is that hybrid retrieval is treated as a first-class database query problem, not an afterthought glued onto ANN. The repo now combines dense vectors, sparse vectors, scalar filters, full-text search, and reranking surfaces in one library. The second strong idea is operational: on-disk features like WAL, file locking, segment management, and an optional DiskANN plugin suggest the team is trying to make embedded retrieval feel production-grade rather than notebook-grade.

The main caution is maturity. The project is still labeled alpha in [`pyproject.toml`](https://github.com/alibaba/zvec/blob/main/pyproject.toml), and the architecture is already broad: C++, pybind, multiple algorithms, plugin loading, FTS, rerankers, and multi-language SDKs. That breadth is exciting, but it also raises the usual young-database questions around consistency, tuning, and long-tail bugs.

## What they built
They built a local vector database library with:

- a C++ collection/storage/query engine in [`src/db/`](https://github.com/alibaba/zvec/tree/main/src/db)
- algorithm implementations grouped under [`src/core/algorithm/`](https://github.com/alibaba/zvec/tree/main/src/core/algorithm)
- Python bindings exposed through [`src/binding/python/binding.cc`](https://github.com/alibaba/zvec/blob/main/src/binding/python/binding.cc)
- a higher-level Python API in [`python/zvec/zvec.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/zvec.py), [`python/zvec/__init__.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/__init__.py), and [`python/zvec/model/collection.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/model/collection.py)
- embedding/reranker extension hooks under [`python/zvec/extension/`](https://github.com/alibaba/zvec/tree/main/python/zvec/extension)

From the README and source together, the product is meant to cover local RAG, embedded search, hybrid retrieval, and app-local document stores without forcing a standalone database server. The current release pitch in [`README.md`](https://github.com/alibaba/zvec/blob/main/README.md) leans on native FTS, hybrid retrieval, and a new on-disk DiskANN index.

## Why it matters
Most vector database products still assume a service boundary: spin up a server, ship embeddings over the wire, and accept operational overhead. zvec pushes the opposite idea. Put retrieval in-process, persist locally, keep latency low, and let the app control the lifecycle.

That matters for agent tools, desktop apps, edge deployments, local RAG, CI diagnostics, and developer tooling where "just run another service" is a tax. It also matters because the repo does not stop at ANN. It tries to make hybrid retrieval a library primitive by combining vector search, FTS, scalar filters, and reranking into one query path.

## Layered architecture dissection
### High-level system shape
The cleanest mental model is:

1. Python or another SDK defines collection schema and query intent.
2. The binding layer forwards those calls into the C++ engine.
3. The collection layer manages files, WAL-ish durability machinery, segments, indexes, and metadata.
4. The SQL/FTS planner turns hybrid query objects into execution plans.
5. Algorithm/index layers search segments and return scored candidates.
6. Python-side extension surfaces can add embeddings or rerankers above the core.

This is visible in the combination of [`python/zvec/zvec.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/zvec.py), [`src/binding/python/binding.cc`](https://github.com/alibaba/zvec/blob/main/src/binding/python/binding.cc), [`src/db/collection.cc`](https://github.com/alibaba/zvec/blob/main/src/db/collection.cc), and [`src/db/sqlengine/sqlengine_impl.cc`](https://github.com/alibaba/zvec/blob/main/src/db/sqlengine/sqlengine_impl.cc).

### Main layers
**1. Packaging and build layer**
[`CMakeLists.txt`](https://github.com/alibaba/zvec/blob/main/CMakeLists.txt) exposes a lot of what the team thinks is important: shared-library targets, Python bindings, C bindings, tools, and optional RaBitQ/DiskANN-related compilation paths. [`pyproject.toml`](https://github.com/alibaba/zvec/blob/main/pyproject.toml) then uses `scikit-build-core` so the Python wheel is really a packaged C++ system, not a pure Python library.

**2. Binding and API layer**
[`src/binding/python/binding.cc`](https://github.com/alibaba/zvec/blob/main/src/binding/python/binding.cc) wires the `_zvec` module and explicitly exports DiskANN plugin controls. On top of that, [`python/zvec/__init__.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/__init__.py) re-exports the public API while auto-registering bundled jieba data for FTS, and [`python/zvec/model/collection.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/model/collection.py) provides the friendlier collection object with DDL/DML/query methods.

**3. Collection and storage layer**
[`src/db/collection.cc`](https://github.com/alibaba/zvec/blob/main/src/db/collection.cc) is the center of gravity. The file includes file locking, version management, delete stores, ID maps, segment helpers, SQL engine access, and write modes for insert/update/upsert/delete. That is where the repo stops feeling like a demo and starts feeling like a database.

**4. Query planning and FTS layer**
[`src/db/sqlengine/sqlengine_impl.cc`](https://github.com/alibaba/zvec/blob/main/src/db/sqlengine/sqlengine_impl.cc) shows the interesting hybrid path. It initializes Arrow compute, registers operators, builds per-segment query plans, detects unsatisfiable filters early, and runs FTS through a tokenizer pipeline plus AST parsing/rewriting. The directory split in [`src/db/sqlengine/analyzer/`](https://github.com/alibaba/zvec/tree/main/src/db/sqlengine/analyzer), [`src/db/sqlengine/parser/`](https://github.com/alibaba/zvec/tree/main/src/db/sqlengine/parser), and [`src/db/sqlengine/planner/`](https://github.com/alibaba/zvec/tree/main/src/db/sqlengine/planner) is exactly the sort of structure you want if hybrid retrieval is going to stay maintainable.

**5. Segment and algorithm layer**
[`src/db/index/segment/segment_manager.cc`](https://github.com/alibaba/zvec/blob/main/src/db/index/segment/segment_manager.cc) manages segment ordering, parallel column operations, and lifecycle changes. Below that, the algorithm families in [`src/core/algorithm/`](https://github.com/alibaba/zvec/tree/main/src/core/algorithm) cover `hnsw`, `ivf`, `vamana`, `flat`, sparse variants, and `diskann`. The split says zvec wants a common DB shell around multiple ANN backends instead of marrying the whole product to one index.

### Request / data / control flow
A normal Python flow starts with [`python/zvec/zvec.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/zvec.py), where `create_and_open()` / `open()` wrap `_Collection` from the compiled extension. The user defines schema and query objects such as [`python/zvec/model/param/query.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/model/param/query.py), then interacts with [`python/zvec/model/collection.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/model/collection.py).

That wrapper ultimately calls the C++ collection implementation in [`src/db/collection.cc`](https://github.com/alibaba/zvec/blob/main/src/db/collection.cc). Inserts and updates flow through `CollectionImpl`, which manages schema prep, file locks, version state, segment allocation, and index-backed writes. Queries fan into [`src/db/sqlengine/sqlengine_impl.cc`](https://github.com/alibaba/zvec/blob/main/src/db/sqlengine/sqlengine_impl.cc), which builds `QueryInfo` per segment, routes FTS expressions through the FTS pipeline, plans execution, and fills the final result set.

The important design choice is that the planner is segment-aware. The engine does not build one global magical query object and hope for the best; it constructs per-segment query info so optimizers can mutate execution independently. That is a real database-engine instinct.

## Important components
**`CollectionImpl` is the operational heart**
[`src/db/collection.cc`](https://github.com/alibaba/zvec/blob/main/src/db/collection.cc) is where file locking, recovery, versioning, ID mapping, delete stores, segment switching, and query entrypoints converge. That is the file I would read first if I wanted to understand whether zvec is serious.

**The SQL engine is the hybrid-retrieval brain**
[`src/db/sqlengine/sqlengine_impl.cc`](https://github.com/alibaba/zvec/blob/main/src/db/sqlengine/sqlengine_impl.cc) does more than parse filters. It coordinates Arrow compute init, operator registration, FTS AST handling, and per-segment planning. This is the difference between "vector DB with some text features" and "query engine that happens to include vectors."

**The pybind module is intentionally thin**
[`src/binding/python/binding.cc`](https://github.com/alibaba/zvec/blob/main/src/binding/python/binding.cc) mostly exposes types and capabilities instead of burying logic in the binding layer. That is the right split. Heavy logic should stay in the core or the higher-level Python facade.

**The extension layer is product strategy, not decoration**
[`python/zvec/extension/`](https://github.com/alibaba/zvec/tree/main/python/zvec/extension) includes OpenAI, Qwen, sentence-transformer, BM25, and reranker helpers. That says the team knows most real users want a batteries-included retrieval workflow, not only a naked ANN index.

## Practical questions and answers
**Is this just a marketing shell around one ANN library?**
No. The source has enough subsystems to count as a small database engine: collection lifecycle, segment management, SQL-ish planning, FTS parsing, multiple algorithm families, bindings, and tests.

**Where does hybrid retrieval actually happen?**
In the planner/query path around [`src/db/sqlengine/sqlengine_impl.cc`](https://github.com/alibaba/zvec/blob/main/src/db/sqlengine/sqlengine_impl.cc) plus the query objects in [`python/zvec/model/param/query.py`](https://github.com/alibaba/zvec/blob/main/python/zvec/model/param/query.py). The architecture is explicitly set up to mix vectors, FTS, and filters.

**What makes it embedded rather than "a DB server in denial"?**
The Python API opens collections directly from a path, the C++ core manages file locks and local persistence, and the whole packaging stack is aimed at producing local libraries and wheels rather than a network daemon.

**What would I copy first?**
The split between core engine, bindings, and ergonomic extension helpers. Also the idea that on-disk ANN features can be optional plugins rather than hard requirements.

**What would I test hard before trusting it in production?**
Crash recovery, compaction/index rebuild edges, hybrid FTS-vector ranking correctness, multi-process read/write semantics, and long-running plugin behavior for DiskANN.

## What is smart
- Treating hybrid retrieval as a planner problem instead of a wrapper script.
- Keeping the binding layer thin and moving real logic into the engine.
- Exposing resource/tuning knobs directly in the Python API.
- Shipping optional external-model integrations in the extension layer instead of pretending users only need raw vectors.
- Making DiskANN a managed plugin surface rather than forcing every platform into the same deployment shape.

## What is flawed or weak
- The repo is already broad for an alpha database: many algorithms, many bindings, and many operational paths.
- The extension layer is convenient, but it risks mixing database concerns with application-API concerns if it keeps growing.
- Embedded databases move ops complexity into the host app. That is often good, but it means app developers inherit tuning and recovery responsibility.
- A repo this young needs especially skeptical benchmarking and correctness testing, even if the README performance graphs look good.

## What we can learn / steal
- Put hybrid retrieval in the core query model, not as an after-market plugin.
- Ship embedded search as a library with serious persistence and recovery semantics.
- Keep the low-level engine language separate from the high-level SDK ergonomics.
- Make heavyweight features like DiskANN pluggable so deployments degrade gracefully.
- Expose operator knobs honestly; hidden heuristics are hard to debug.

## How we could apply it
If we wanted an app-local retrieval store, this repo suggests a good blueprint:

1. keep the storage/query core in one fast language,
2. expose thin bindings,
3. provide a small, opinionated high-level SDK,
4. support hybrid query types from day one,
5. make optional heavyweight backends pluggable,
6. bias for file-path-based local operation rather than service sprawl.

For agent systems specifically, zvec's embedded shape is appealing. A local codebase assistant, evaluation harness, or desktop knowledge tool often wants retrieval without another container to babysit.

## Bottom line
zvec is one of the more interesting trending infra repos of the day because it is trying to make embedded retrieval feel like a real database product, not a toy ANN wrapper.

The reusable lesson is simple: if local-first retrieval is the goal, take the database part seriously. zvec does that by combining collection lifecycle, segment storage, FTS, ANN algorithms, and bindings into one coherent stack.
