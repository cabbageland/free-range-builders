# Beads

- Repo: `gastownhall/beads`
- URL: <https://github.com/gastownhall/beads>
- Date: 2026-04-27
- Repo snapshot studied: `main` @ `674561360d7cd98a4e00fb5993bec4720375b043`
- Why picked today: It was genuinely hot, AI-adjacent without being another thin wrapper, and the implementation is unusually systems-minded. It is trying to solve a real agent problem, persistent coordination and memory, with a versioned database instead of markdown theater.

## Executive summary

Beads is a task graph and memory system for coding agents, but the interesting part is not the CLI surface. The interesting part is the storage model. It treats agent coordination as shared structured state, then uses Dolt as the versioned data plane under that state.

The repo is strongest where it refuses the usual "agent memory" hand-waving and instead builds concrete machinery for IDs, dependencies, compaction, sync, routing, ephemeral messages, and multi-project isolation. The core design move is: keep the user-facing contract small (`bd create`, `bd ready`, `bd show`), while hiding a lot of storage and workflow discipline underneath.

The weakness is that the repository is sprawling. The command surface is huge, the amount of integration logic is high, and the implementation burden of supporting embedded Dolt, server Dolt, MCP, git hooks, Claude integration, redirects, and ephemeral issue modes makes the codebase feel closer to an operating system than a simple CLI.

1. task tracking with dependencies and hierarchy
2. persistent agent memory and workflow context
3. multi-agent coordination through shared structured state
4. sync/versioning via Dolt instead of plain files

## Why it matters

Most agent workflows still collapse into markdown TODO lists, ad hoc chat memory, or issues with no real dependency semantics. Beads is a more serious answer: use a database with version control semantics, then design agent workflows around graph state instead of prose.

That matters because the hard part of agents is not usually "can they call a model". It is whether they can preserve shared state, see each other's work, recover after context loss, and avoid coordination drift. Beads is explicitly designed around those failure modes.

## Layered architecture dissection

### High-level system shape

Beads is basically a layered coordination stack:

1. a CLI command layer accepts human or agent intents
2. workspace-discovery and routing logic finds the active `.beads` context
3. a storage interface chooses server-mode Dolt or embedded Dolt
4. transaction logic writes issue graph state, metadata, wisps, labels, and dependencies
5. integrations project that state back into agent workflows through hooks, MCP, and setup commands

The smart part is that the agent UX is thin. Most of the real value lives below the CLI in routing and state-management machinery.

### Main layers

#### 1. Command and UX layer

The command tree lives in <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd> and is massive.

<https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/main.go> wires global flags, command registration, environment loading, read-only command handling, telemetry, hook runners, and store lifecycle. This file makes it obvious that `bd` is not just a simple CLI parser. It is the session coordinator for the whole system.

The command surface is split into many focused files, for example:

- task creation and updates in <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/create.go> and <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/update.go>
- dependency graph operations in <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/dep.go> and <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/graph.go>
- workflow-context injection in <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/prime.go>
- agent/tool setup in <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/setup>

#### 2. Workspace resolution and shared-state routing layer

A surprisingly important layer is <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/internal/beads/beads.go>.

This code handles `.beads` discovery, redirect following, stable worktree preference, and shared database identity preservation. That sounds boring until you remember the product lives inside messy real repos, worktrees, branches, sandboxes, and agent sessions. This layer is what stops the system from writing to the wrong project state.

This is one of the repo's best hidden insights: if agent memory is tied to workspace identity, path resolution and redirect handling are first-class architecture, not incidental utility code.

#### 3. Storage abstraction layer

The interfaces and shared storage contracts live under <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage>.

This layer defines the model as operations over issues, dependencies, labels, compaction, metadata, and version-control behaviors, rather than leaking a single DB implementation everywhere. It is what lets the project support both an external Dolt sql-server path and an embedded engine path.

#### 4. Server-mode Dolt layer

The main multi-writer implementation is in <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage/dolt/store.go> and <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage/dolt/transaction.go>.

This is where the repo becomes genuinely interesting. The transaction code pins one SQL connection for the whole operation so the SQL transaction and the later `DOLT_COMMIT` operate on the same Dolt session. The code comments are very explicit about the failure mode: if you mix connections, Dolt can commit the wrong working set.

That is the architectural heart of the system. Beads is not just using Dolt as a dumb persistence backend. It is shaping its concurrency model around Dolt's session and commit semantics.

The design note in <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/docs/design/dolt-concurrency.md> makes the intent even clearer: they retired branch-per-worker thinking and moved toward all-on-main transaction discipline so agents see shared state continuously.

#### 5. Embedded Dolt layer

The single-writer local mode lives in <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage/embeddeddolt/store.go> and <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage/embeddeddolt/open.go>.

The notable implementation choice here is aggressive safety around locking and short-lived connections. The store acquires a filesystem lock for its lifetime to prevent concurrent embedded-engine initialization panics, then opens transactional connections briefly to minimize contention. That is a very practical systems trade: simpler deployment, but only by taking locking discipline seriously.

#### 6. Ephemeral messaging and mixed-lifecycle issue layer

A distinctive subsystem lives in <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage/dolt/ephemeral_routing.go>.

Beads separates durable issues from ephemeral "wisps" and includes routing logic that decides which table an ID should hit, even for explicit IDs that do not match the normal ephemeral naming pattern. This is smart because agent coordination often needs both kinds of state: real long-lived work items and short-lived coordination artifacts.

#### 7. Agent integration layer

There are several agent-facing surfaces:

- MCP server in <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/integrations/beads-mcp>
- Claude Code setup and commands in <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/integrations/claude-code>
- workflow-context injection in <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/prime.go>

The README for beads-mcp explicitly argues that shell access plus hooks is cheaper and lower-latency than MCP for environments that permit it. That is a very grounded product choice. They are not fetishizing protocols. They are optimizing for token budget and operational fit.

### Request / data / control flow

A typical write flow looks like this:

1. a user or agent runs a command like `bd create` through <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/main.go>
2. workspace resolution code in <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/internal/beads/beads.go> finds the right `.beads` context and any redirect target
3. storage setup selects either <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage/dolt> or <https://github.com/gastownhall/beads/tree/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage/embeddeddolt>
4. the write runs inside a storage transaction, with table routing for issues versus wisps handled in <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage/dolt/transaction.go> and <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/internal/storage/dolt/ephemeral_routing.go>
5. Dolt version-control operations stage and commit the modified tables
6. hooks, MCP, or agent setup surfaces expose the updated state back to tools and sessions

The prime/context flow is separate but complementary:

1. `bd prime` in <https://github.com/gastownhall/beads/blob/674561360d7cd98a4e00fb5993bec4720375b043/cmd/bd/prime.go> detects whether MCP is active
2. it loads local or global `PRIME.md` overrides when present
3. it emits compact workflow instructions into the agent session so the tool usage pattern survives compaction or session restarts

That is another quiet but important systems move: memory is not only data storage, it is also workflow rehydration.

## Practical questions and answers

**What problem is Beads actually solving?**

Persistent structured coordination for agents. Not just remembering facts, but tracking work, dependencies, readiness, ownership, and state transitions in a form agents can query reliably.

**Why use Dolt instead of SQLite or markdown?**

Because they want version-controlled shared state, sync semantics, branching history, and multi-writer behavior. Dolt gives them database operations plus version-control primitives in one place.

**What is the key architectural trick?**

Pinning writes to a single Dolt session and treating commit semantics as first-class. The repository clearly learned that database correctness here depends on how Dolt sessions and working sets behave, not just on issuing SQL.

**Why have both server mode and embedded mode?**

Because the product is serving two different realities. Embedded mode is low-friction local use. Server mode is for concurrent multi-agent writing. The codebase pays complexity to cover both.

**What are wisps buying them?**

A second lifecycle. Some coordination artifacts should not behave like durable project tasks. The explicit routing for ephemeral state keeps the durable task graph cleaner.

**Why does `bd prime` matter so much?**

Because agent failure is often context loss, not storage loss. Re-injecting workflow instructions into sessions is part of the memory system, even though it looks like a convenience command.

## What is smart

- The repo treats workspace discovery and redirect handling as serious architecture, not glue code.
- The Dolt transaction design is honest about the storage engine's real semantics and codes around them carefully.
- The product separates durable issues from ephemeral wisps, which matches how agent work actually behaves.
- The MCP README's recommendation of CLI plus hooks over MCP when shell access exists is refreshingly pragmatic.
- The `bd prime` mechanism is a clever answer to agent context compaction. It is memory as rehydrated operating procedure, not only stored facts.

## What is flawed or weak

- The command tree is enormous, which raises the maintenance and discoverability cost.
- Supporting embedded Dolt, server Dolt, MCP, hooks, plugins, redirects, and lots of integrations creates a very wide blast radius for regressions.
- Some of the most important ideas are buried in code comments and design docs rather than made obvious from the main README.
- The repo feels operationally heavy for teams that just need lightweight task tracking. This is a serious system, not a tiny utility.

## What we can learn / steal

- If agent coordination matters, model state structurally and queryably, not as markdown blobs.
- Treat workspace identity and routing as core design problems in multi-repo or multi-worktree tooling.
- When using an unusual storage substrate, expose its real concurrency model in code and docs instead of pretending the backend is generic.
- Design for mixed-lifecycle artifacts. Not everything should be a durable task.
- Reinjection of workflow context is worth productizing. Agents forget procedures even when the data still exists.

## How we could apply it

For our own agent systems, the most reusable move is not necessarily Dolt itself. It is the combination of:

- a small stable command surface
- structured dependency-aware work items
- explicit ephemeral versus durable state
- session rehydration hooks
- careful workspace routing

If we copied only one idea, I would copy the "shared data plane for agents" mindset. If we copied two, I would also copy the `prime` style workflow rehydration.

## Bottom line

Beads is one of the more serious agent-infrastructure repos on the current trending page. Under the marketing line about memory, it is really a coordination database with strong opinions about agent workflow.

The repo's main insight is that long-horizon agents need a shared, versioned state machine more than they need more vibes. The repo's main risk is complexity creep from trying to be CLI, database layer, sync engine, MCP server, hook system, and agent integration platform all at once.

Still, there is a lot worth stealing here.
