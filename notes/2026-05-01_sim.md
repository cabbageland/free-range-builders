# Sim

- Repo: `simstudioai/sim`
- URL: https://github.com/simstudioai/sim
- Date: 2026-05-01
- Repo snapshot studied: `main@ecbf5e5eb190088a5ab7f9766d748715e9862e5b`
- Why picked today: It is AI-related, currently trending, and more interesting than the average agent-workflow repo because it is not just a prompt shell. It is a very large product monorepo that tries to turn “AI agents” into a real application platform, with workflow editing, execution, realtime collaboration, MCP exposure, hosted connectors, and self-hosting paths.

## Executive summary
Sim is an open-source agent-workflow platform, basically a visual automation builder for LLM-native workflows. The user-facing story is canvas-based agent building, but the real repo is a full-stack monorepo: Next.js app, Socket.IO realtime service, normalized workflow persistence, a very large tool/block catalog, provider adapters, background execution machinery, MCP sync, and SDK/CLI packaging.

The strongest insight is that Sim is not architected like “one smart agent.” It is architected like a workflow operating system. The center of gravity is not prompts, it is schema, persistence, orchestration, execution state, and a giant registry surface that makes workflows composable. The cleanest proof is the chain from [`packages/workflow-persistence/src/save.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/workflow-persistence/src/save.ts) and [`packages/workflow-persistence/src/load.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/workflow-persistence/src/load.ts) into [`apps/sim/executor/dag/builder.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor/dag/builder.ts) and then the runtime in [`apps/sim/executor/execution/engine.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor/execution/engine.ts).

## What they built
They built a platform for designing, deploying, and running AI workflows.

At a high level, the product includes:
- a web app for editing and operating workflows on a visual canvas in [`apps/sim/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim)
- a separate realtime collaboration and event service in [`apps/realtime/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/realtime)
- a workflow execution engine in [`apps/sim/executor/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor)
- an enormous block and tool catalog in [`apps/sim/blocks/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/blocks) and [`apps/sim/tools/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/tools)
- shared persistence, auth, security, SDK, and protocol packages in [`packages/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages)
- self-hosting entrypoints via [`packages/cli/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/cli), Docker files, and Helm charts

This is not just an “agent app.” It is an attempt to package agent construction, execution, and integration sprawl into a product platform.

## Why it matters
Most agent repos are thin wrappers around model calls plus a pretty README. Sim matters because it attacks the actual hard part: making agent workflows persistent, collaborative, configurable, extensible, and operable.

That makes it useful to study for:
- workflow-product architecture
- monorepo boundary design
- translating visual nodes into executable graphs
- keeping huge integration surfaces organized
- exposing internal workflows back out as MCP tools

## Layered architecture dissection
### High-level system shape
The system shape is: visual workflow editor -> normalized workflow state -> execution graph runtime -> integrations/tool surface -> persistence/logging/realtime layers around it.

The crucial choice is that Sim separates three concerns that many agent products mush together:
- canvas authoring model
- executable workflow graph
- integration catalog

That separation is visible in the split among [`apps/sim/blocks/registry.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/blocks/registry.ts), [`apps/sim/executor/execution/engine.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor/execution/engine.ts), and [`packages/workflow-persistence/src/index.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/workflow-persistence/src/index.ts).

### Main layers
**1. Product shell and route layer**
- [`apps/sim/app/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/app)
- [`apps/sim/components/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/components)
- [`apps/sim/stores/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/stores)

This is the user-facing shell: auth, workspace UI, workflow canvas, settings, logs, docs links, and client-side state. It is big because the product is big.

**2. Block-definition layer**
- [`apps/sim/blocks/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/blocks)
- especially [`apps/sim/blocks/registry.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/blocks/registry.ts)

This layer defines what a workflow can be made of. The registry file is enormous because each block is treated as a first-class product primitive. That is both a strength and a maintenance hazard.

**3. Execution-runtime layer**
- [`apps/sim/executor/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor)
- [`apps/sim/executor/execution/engine.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor/execution/engine.ts)
- [`apps/sim/lib/workflows/executor/execute-workflow.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/lib/workflows/executor/execute-workflow.ts)

This is where canvas state becomes live work. [`apps/sim/executor/dag/builder.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor/dag/builder.ts) compiles user workflows into a DAG with explicit loop/parallel sentinel structure, and [`apps/sim/executor/execution/engine.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor/execution/engine.ts) manages ready queues, concurrent execution, cancellation, pause/resume snapshots, final outputs, and orchestration callbacks. This is the real product core.

**4. Workflow-lifecycle and persistence layer**
- [`apps/sim/lib/workflows/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/lib/workflows)
- [`packages/workflow-persistence/src/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/workflow-persistence/src)
- [`packages/db/schema.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/db/schema.ts)

This layer persists workflows into normalized tables like [`workflow`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/db/schema.ts), [`workflow_blocks`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/db/schema.ts), [`workflow_edges`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/db/schema.ts), and subflow records via [`packages/workflow-persistence/src/save.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/workflow-persistence/src/save.ts). It then reconstructs executable state from that normalized representation in [`packages/workflow-persistence/src/load.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/workflow-persistence/src/load.ts). This is much more serious than keeping the whole canvas as one opaque JSON blob.

**5. Integration and provider layer**
- [`apps/sim/tools/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/tools)
- [`apps/sim/connectors/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/connectors)
- [`apps/sim/providers/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/providers)

This is the monetizable surface area: lots of providers, lots of tools, lots of external systems. The repo’s scale is driven less by model cleverness than by integration sprawl.

**6. Realtime collaboration layer**
- [`apps/realtime/src/index.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/realtime/src/index.ts)
- [`apps/realtime/src/handlers/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/realtime/src/handlers)
- [`apps/realtime/src/rooms/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/realtime/src/rooms)

Sim split realtime into a dedicated service, with Redis-backed multi-pod support when configured. That is a product-company move, not a demo-repo move.

**7. Protocol and externalization layer**
- [`apps/sim/lib/mcp/workflow-mcp-sync.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/lib/mcp/workflow-mcp-sync.ts)
- [`packages/realtime-protocol/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/realtime-protocol)
- [`packages/ts-sdk/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/ts-sdk)
- [`packages/python-sdk/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/python-sdk)

The clever extension is that workflows can become outward-facing tools via MCP sync. That means the internal workflow system is also an external capability surface.

### Request / data / control flow
A typical path looks like this:
1. a user edits a workflow in the web app under [`apps/sim/app/workspace/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/app/workspace)
2. block definitions from [`apps/sim/blocks/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/blocks) describe the allowed nodes and interfaces
3. workflow state is saved through normalized persistence helpers in [`packages/workflow-persistence/src/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/workflow-persistence/src) into DB tables defined in [`packages/db/schema.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/db/schema.ts)
4. execution starts through [`apps/sim/lib/workflows/executor/execute-workflow.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/lib/workflows/executor/execute-workflow.ts)
5. the runtime schedules nodes through [`apps/sim/executor/execution/engine.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor/execution/engine.ts), including cancellation and pause handling
6. blocks call tools/providers from [`apps/sim/tools/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/tools) and [`apps/sim/providers/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/providers)
7. collaboration and status events can flow through the socket service in [`apps/realtime/`](https://github.com/simstudioai/sim/tree/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/realtime)
8. deployed workflows can be projected outward into MCP tools via [`apps/sim/lib/mcp/workflow-mcp-sync.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/lib/mcp/workflow-mcp-sync.ts)

The big architectural pattern is author once, persist structurally, execute as a graph, then optionally publish as a tool.

## Important components
**Block registry as product contract**
[`apps/sim/blocks/registry.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/blocks/registry.ts) is effectively the product catalog. It is where “what users can build” becomes explicit code. You can see the shape of a first-class block in [`apps/sim/blocks/blocks/agent.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/blocks/blocks/agent.ts), which bundles UI config, model/provider selection, tool access, memory options, and output schema in one definition.

**DAG builder plus ExecutionEngine**
[`apps/sim/executor/dag/builder.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor/dag/builder.ts) and [`apps/sim/executor/execution/engine.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/executor/execution/engine.ts) together are the technical heart. The code shows mature concerns: graph compilation, loop/parallel validation, concurrency, cancellation polling, abort-signal reuse, pausing, resuming, and incomplete-log finalization.

**Normalized workflow persistence**
[`packages/workflow-persistence/src/index.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/packages/workflow-persistence/src/index.ts) matters because it means they do not treat workflows as disposable front-end state. They treat them as a durable graph data model.

**Realtime sidecar**
[`apps/realtime/src/index.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/realtime/src/index.ts) makes collaboration and room state a deployable subsystem instead of an afterthought bolted into the Next.js app.

**Workflow-to-MCP bridge**
[`apps/sim/lib/mcp/workflow-mcp-sync.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/lib/mcp/workflow-mcp-sync.ts) is one of the smartest components in the repo, and [`apps/sim/lib/mcp/service.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/lib/mcp/service.ts) shows the operational side: discovery, cacheing, SSRF/domain checks, retry logic, and connection-state bookkeeping. This is not toy MCP support. It turns internal workflows into externally consumable tool surfaces by deriving parameter schema from workflow input blocks.

## Practical questions and answers
**What is the real product here?**
A workflow platform, not a single agent. The repo is optimized around authoring, running, observing, and reusing graph-shaped automations.

**What is the key architectural bet?**
That agent products become useful when workflows are explicit, persistent, and integration-rich. Sim bets on workflow infrastructure more than on one-off prompt magic.

**Why the giant registry surface?**
Because the business value is breadth. Every connector, provider, and tool becomes another workflow building primitive. The codebase reflects product-surface accumulation.

**Where would this hurt in production?**
Registry sprawl, test burden, dependency churn, and execution edge cases. A platform with thousands of integration permutations gets expensive to keep coherent.

**What should a builder copy?**
The separation between authoring model, execution runtime, persistence model, and outward protocol surface. That is the strongest architectural lesson here.

## What is smart
- Treating workflows as a normalized relational model, not just serialized canvas JSON, then compiling them into an execution DAG instead of trying to run raw editor state directly.
- Splitting realtime into its own service for multi-user collaboration.
- Building a real execution engine with pause/resume and cancellation instead of a naive sequential runner.
- Turning deployed workflows into MCP tools, which lets the platform expose its own internals as reusable external capabilities.
- Adding explicit monorepo-boundary checks in [`scripts/check-monorepo-boundaries.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/scripts/check-monorepo-boundaries.ts), a good sign that the team knows architecture entropy is a real threat.

## What is flawed or weak
- The repo is huge, and the sheer number of tools and blocks creates obvious maintenance drag.
- [`apps/sim/blocks/registry.ts`](https://github.com/simstudioai/sim/blob/ecbf5e5eb190088a5ab7f9766d748715e9862e5b/apps/sim/blocks/registry.ts) is impressive, but it also signals a scaling risk: giant registries are easy to grow and hard to keep conceptually clean.
- The product surface seems broad enough that quality could vary wildly across integrations.
- “Agent platform” positioning always risks overselling autonomy when the real engineering value is workflow plumbing and integration management.
- Self-hosting is supported, but the dependency surface is nontrivial enough that operational complexity is clearly real.

## What we can learn / steal
- Build workflow systems around explicit graph primitives and durable persistence.
- Separate UI authoring state from runtime execution state.
- Treat tool and provider catalogs as platform infrastructure, not incidental helpers.
- Make collaboration and protocol surfaces first-class if the product is team-oriented.
- Add structural guardrails before a monorepo turns to mush.

## How we could apply it
If we were building an internal automation product, I would borrow the shape more than the branding:
- visual or structured authoring layer
- normalized workflow model in the database
- execution runtime with pause/cancel/resume
- separate realtime subsystem if collaboration matters
- optional publication layer that exposes internal workflows as tools or APIs

The standout reusable idea is the workflow-to-MCP bridge. That is a strong pattern for turning private automation into composable capability.

## Bottom line
Sim is worth studying because it treats agent workflows like a software platform, not a demo. The core lesson is that the durable value is not “AI agents on a canvas.” It is the infrastructure underneath: block contracts, normalized workflow persistence, execution control, realtime coordination, and a clean path from internal workflow to external tool.

I would copy the workflow-system architecture. I would be wary of the maintenance tax created by a giant integration and block catalog.
