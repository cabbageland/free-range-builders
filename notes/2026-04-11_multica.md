# Multica

- Repo: `multica-ai/multica`
- URL: https://github.com/multica-ai/multica
- Date: 2026-04-11
- Repo snapshot studied: `main` @ `f9f061de4ce5c2b77274d01bd576152e4059acc7`
- Why picked today: It is hot right now, AI-native in a non-fake way, and the implementation is interesting because it treats “agent coworker” as an orchestration problem spanning board UI, realtime coordination, daemonized local runtimes, and resumable execution environments.

## Executive summary

Multica is not just another “run an agent from a web page” wrapper. The interesting thing here is that it tries to make agents behave like durable teammates inside a task system: assign work, queue it, dispatch it to a runtime, stream execution back live, preserve the workdir/session for follow-up tasks, and turn the result into comments, status changes, and reusable skills.

The repo’s real center of gravity is the seam between the Go backend and the local daemon. The web app is mostly a polished control surface. The backend owns task state, auth, workspace scoping, and event broadcast. The daemon owns agent execution, runtime registration, repo cache syncing, isolated workdirs, session resume, and usage reporting. That split is the thing to study.

The strongest idea in the repo is that agent work should be **stateful across tasks**, not a fresh sandbox every time. The daemon preserves prior workdirs and prior session IDs, which means an issue can accumulate context like a real coworker thread instead of resetting on every assignment. The main weakness is that the product promise is broader than the implementation maturity: once you support multiple agent vendors, local daemons, cloud/self-hosting, live streaming, repo cache, skill injection, and issue/chat modes, operational edge cases multiply fast.

## What they built

They built an open-source managed-agents platform with four main product surfaces:

- a web app for issues, projects, agents, runtimes, inbox, and skills
- a Go backend API with realtime events and task orchestration
- a local CLI + daemon that registers machine-local agent runtimes and executes tasks
- a shared monorepo package layer for frontend state, API clients, reusable UI, and feature views

At a practical level, the system lets a team:

- create workspaces and projects
- create agents tied to runtimes/providers
- assign issues to those agents
- have a local daemon claim and run the work via Claude Code / Codex / OpenClaw / OpenCode
- stream progress and tool activity back into the app
- preserve workdir and session continuity for follow-up tasks
- manage reusable skills and runtime health/update operations

## Why it matters

A lot of “AI teammate” products are really just glorified prompt launchers. Multica is more ambitious and more infrastructural. It is trying to answer a harder question: what does the software stack look like if agents are first-class actors inside a team workflow rather than one-shot chat completions?

That matters because the difficult problems are not just inference quality. They are:

- state handoff between tasks
- runtime discovery and health
- assignment and queue semantics
- workspace isolation
- comment/thread integration
- live visibility into what the agent is doing
- durability across restarts and updates

Multica is interesting precisely because it tackles those boring-but-real product constraints.

## Layered architecture dissection

### High-level system shape

The architecture is roughly:

1. **Frontend views** render issues/agents/projects/runtimes and subscribe to realtime updates.
2. **Frontend core package** owns typed API queries/mutations, local stores, and websocket sync.
3. **Go backend** exposes authenticated REST-ish APIs plus websocket events.
4. **Task service** turns issue/chat actions into queued agent tasks and reconciles state transitions.
5. **Local daemon** registers available providers, claims tasks, prepares execution environments, runs external agent CLIs, and reports outputs/usage back.
6. **Postgres** holds all the durable truth: tasks, comments, agents, runtimes, workspaces, skills, usage, inbox, etc.

So the product is not “frontend talking directly to LLM vendor APIs.” It is a proper task system with a runtime control plane.

### Main layers

#### 1. Presentation layer

The main web shell lives in [`apps/web/`](https://github.com/multica-ai/multica/tree/main/apps/web), with route groups like:

- [`app/(dashboard)/issues`](https://github.com/multica-ai/multica/tree/main/apps/web/app/%28dashboard%29/issues)
- [`app/(dashboard)/agents`](https://github.com/multica-ai/multica/tree/main/apps/web/app/%28dashboard%29/agents)
- [`app/(dashboard)/runtimes`](https://github.com/multica-ai/multica/tree/main/apps/web/app/%28dashboard%29/runtimes)
- [`app/(dashboard)/settings`](https://github.com/multica-ai/multica/tree/main/apps/web/app/%28dashboard%29/settings)

The interesting design choice is that most product UI is *not* stuffed directly into the app routes. It is pushed into reusable feature packages, which keeps the Next app mostly as routing/composition glue.

#### 2. Frontend domain/state layer

[`packages/core/`](https://github.com/multica-ai/multica/tree/main/packages/core) is the frontend brain. It contains:

- API client and websocket client: [`api/client.ts`](https://github.com/multica-ai/multica/blob/main/packages/core/api/client.ts), [`api/ws-client.ts`](https://github.com/multica-ai/multica/blob/main/packages/core/api/ws-client.ts)
- issue queries/mutations/stores: [`issues/`](https://github.com/multica-ai/multica/tree/main/packages/core/issues)
- project, workspace, runtime, inbox, chat, pin modules as separate domain folders
- realtime provider/hooks in [`realtime/`](https://github.com/multica-ai/multica/tree/main/packages/core/realtime)

That is a good split. It prevents route-level UI code from owning transport and synchronization logic.

The websocket client in [`packages/core/api/ws-client.ts`](https://github.com/multica-ai/multica/blob/main/packages/core/api/ws-client.ts) is intentionally simple: token + workspace-scoped connection, typed event handlers, auto-reconnect, reconnect callbacks. Nothing fancy, but it matches the product need.

#### 3. Reusable view layer

[`packages/views/`](https://github.com/multica-ai/multica/tree/main/packages/views) is where most actual product surfaces live:

- [`issues/components/`](https://github.com/multica-ai/multica/tree/main/packages/views/issues/components)
- [`agents/components/`](https://github.com/multica-ai/multica/tree/main/packages/views/agents/components)
- [`runtimes/components/`](https://github.com/multica-ai/multica/tree/main/packages/views/runtimes/components)
- [`skills/components/`](https://github.com/multica-ai/multica/tree/main/packages/views/skills/components)
- [`chat/components/`](https://github.com/multica-ai/multica/tree/main/packages/views/chat/components)

This is a pretty clean monorepo habit: **state in core, presentational/feature assemblies in views, primitive widgets in ui**.

#### 4. Backend API and middleware layer

The backend entry surface is the Chi router in [`server/cmd/server/router.go`](https://github.com/multica-ai/multica/blob/main/server/cmd/server/router.go).

That file tells you the whole system shape quickly:

- public auth endpoints
- daemon-only/runtime endpoints under `/api/daemon`
- protected user/workspace endpoints for issues, projects, pins, agents, skills, runtimes, usage, chat, and inbox
- websocket endpoint at `/ws`

This is one of the most important files in the repo because it makes the separation explicit: **human-facing product API** and **daemon control/reporting API** coexist, but they are not the same thing.

#### 5. Task orchestration layer

The real business logic center is [`server/internal/service/task.go`](https://github.com/multica-ai/multica/blob/main/server/internal/service/task.go).

That layer handles:

- enqueue issue-triggered tasks
- enqueue mention-triggered tasks
- enqueue chat tasks
- runtime/agent-aware claiming
- start/complete/fail/cancel transitions
- agent status reconciliation
- broadcast of task and chat events
- creation of agent comments after completion/failure

The key thing here is that the system models agent work as a proper queue with lifecycle transitions, not an ad hoc “run provider call now” button.

#### 6. Runtime execution layer

This is the most interesting subsystem in the repo.

The daemon lives primarily in [`server/internal/daemon/daemon.go`](https://github.com/multica-ai/multica/blob/main/server/internal/daemon/daemon.go), with CLI control in [`server/cmd/multica/cmd_daemon.go`](https://github.com/multica-ai/multica/blob/main/server/cmd/multica/cmd_daemon.go).

What it does:

- loads CLI auth/config
- registers runtimes for watched workspaces
- syncs workspace repos into a local cache
- heartbeats to the server
- polls/claims pending tasks by runtime
- prepares isolated execution environments
- injects runtime-specific config/skills/context
- executes provider CLIs with per-task env vars
- streams tool/text/thinking back to the backend
- preserves workdir + session ID for later task reuse
- reports usage and supports update/ping flows

This daemon is the real product moat. Anyone can ship a dashboard. Fewer projects build a decent local-runtime agent runner.

### Request / data / control flow

A typical issue-assignment flow looks like this:

1. User assigns an issue to an agent from the UI.
2. Backend task service creates a queued task via [`server/internal/service/task.go`](https://github.com/multica-ai/multica/blob/main/server/internal/service/task.go).
3. A local daemon heartbeat/poll loop in [`server/internal/daemon/daemon.go`](https://github.com/multica-ai/multica/blob/main/server/internal/daemon/daemon.go) claims the task for a matching runtime.
4. The daemon marks it started, prepares or reuses an execution workdir, constructs the agent prompt/context, and launches the provider backend.
5. Tool usage, text, and thinking are batched and reported back to the server.
6. The backend broadcasts task events over websocket; the frontend updates issue/live-task UI.
7. On completion, the daemon reports output, usage, session ID, and workdir.
8. The backend persists the result and may materialize the output as an agent comment on the issue.

That is a solid flow. It cleanly separates orchestration, execution, and presentation.

## Important components

### Router as architecture manifest

[`router.go`](https://github.com/multica-ai/multica/blob/main/server/cmd/server/router.go) is not just HTTP plumbing. It is the architecture manifest. You can see the product capability matrix directly from the route groups: issues, projects, agents, skills, runtimes, inbox, chat, daemon registration, runtime health, usage, websocket.

### `TaskService`

[`TaskService`](https://github.com/multica-ai/multica/blob/main/server/internal/service/task.go) is where issue assignment stops being CRUD and becomes orchestration. It handles enqueueing, claiming, cancellation, completion, agent comment generation, and realtime event publishing.

### The daemon

[`Daemon`](https://github.com/multica-ai/multica/blob/main/server/internal/daemon/daemon.go) is the system’s execution engine. It manages watched workspaces, registered runtimes, polling fairness, concurrency limits, agent execution, and task streaming.

### Realtime websocket client

[`WSClient`](https://github.com/multica-ai/multica/blob/main/packages/core/api/ws-client.ts) is deliberately boring in a good way. It keeps connection state simple and event-driven instead of embedding business logic in the transport layer.

### View packages

The feature UIs in [`packages/views/`](https://github.com/multica-ai/multica/tree/main/packages/views) are the product’s day-to-day face. The issue board, runtime panels, and skills browser make the backend/daemon machinery legible to humans.

## Practical questions and answers

### What is the smartest idea here?

The daemon reuses prior workdirs and prior session IDs for the same issue/agent path. That is a much better model than pretending every task is stateless. It gives agents something closer to continuity.

### What assumption does the system make?

It assumes local runtimes are trustworthy enough and stable enough to be part of the orchestration loop. That is great for self-hosting and power users, but it means reliability depends on daemon health, local CLI correctness, and environment drift.

### Where does it likely fail in production?

At boundaries:

- daemon auth/config drift
- runtime registration edge cases
- long-running task cancellation/resume weirdness
- mismatches between provider-specific behavior and the common abstraction
- local environment differences across machines

### Is the repo structurally clean?

Yes, mostly. The package boundaries are more disciplined than many hot repos. The biggest complexity concentration is justifiably in the daemon and backend task machinery.

### Is any part mostly hype?

The marketing line about your next hires not being human is obviously spicier than the actual software. The real value is much more grounded: it is a decent local-runtime agent orchestration stack.

## What is smart

- The backend/daemon split is correct. Execution belongs near the local CLI/runtime, not inside the web app.
- The monorepo layering is disciplined: apps vs core vs views vs ui.
- Task lifecycle is modeled explicitly instead of as magical “agent job” state.
- Websocket syncing is thin and composable rather than over-engineered.
- Workdir/session reuse is genuinely important and easy to underestimate.
- Runtime ping/update/usage flows suggest they are thinking operationally, not just demo-first.
- The design system doc in [`docs/design.md`](https://github.com/multica-ai/multica/blob/main/docs/design.md) is opinionated in a useful way; product polish is being systematized, not improvised.

## What is flawed or weak

- The promise surface is broad. Multi-provider execution + daemon orchestration + board UX + skills + chat + self-hosting is a lot to keep reliable.
- The architecture is only as portable as the agent CLIs beneath it; vendor quirks will leak through.
- Preserving workdirs is powerful, but it also creates more cleanup/state-corruption/debugging cases.
- The codebase is still early enough that some complexity feels ahead of proven product necessity.
- “Reusable skills” is compelling, but these systems often struggle to keep skill libraries from turning into stale cargo-cult blobs.

## What we can learn / steal

- Treat agent execution as a **runtime control plane**, not as a button in a chat box.
- Keep the web app thin and push real orchestration into backend + daemon layers.
- Preserve stateful continuity for repeated work on the same task.
- Separate frontend primitives, domain state, and feature views in the monorepo.
- Make realtime transport simple; put meaning in events and state transitions instead.
- Build operational surfaces early: health, usage, update, status, logs.

## How we could apply it

If we were building our own agent-work system, the first thing worth copying is the **task/daemon/workdir** model:

- queue durable tasks
- claim them from a runtime-bound worker
- keep a reusable workdir/session per issue
- stream events back to the UI
- persist usage and execution metadata

I would copy that before copying any of the teammate-branding language.

For a smaller product, I would probably simplify in this order:

1. support one provider first
2. keep the daemon but reduce surface area
3. ship issue/task continuity before fancy social metaphors
4. make skills extremely narrow and explicit

## Bottom line

Multica is interesting because it is trying to operationalize agents as coworkers, not just invoke them. The repo is worth studying less for the UI pitch and more for the runtime model: backend-owned task semantics, daemon-owned execution, websocket-visible progress, and preserved workdir/session continuity.

That continuity piece is the keeper. A lot of agent products still act like every job starts from amnesia. Multica is one of the more concrete attempts to treat repeated agent work like an ongoing engineering thread instead of a pile of disconnected prompts.
