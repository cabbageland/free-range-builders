## Executive summary

This repo is not a software product in the normal sense. It is a curated prompt-programming toolkit for coding agents, packaged as composable skills instead of one giant methodology. The core move is smart: instead of trying to replace engineering process with a monolith, it decomposes repeated agent failure modes into small, inspectable interventions like better planning, TDD discipline, architecture review, and domain-language capture.

The repo is strongest where it behaves like a policy lattice. The top-level <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/README.md>, plugin manifest at <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/.claude-plugin/plugin.json>, repo contract in <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/CLAUDE.md>, and per-skill `SKILL.md` files all reinforce the same design: keep skills small, explicit, and composable.

The main weakness is also the main tradeoff. This system depends on careful curation and duplicated metadata. Every skill has to be documented in multiple places, and quality control is mostly social plus repo conventions. That is manageable now because the repo is small, but it could get fragile as the catalog grows.

## Why it matters

Most agent repos either chase full autonomy or ship a grab bag of prompts. This repo takes a better middle path.

It assumes:
- engineering process still matters
- agents fail in repeatable ways
- those failures can be countered with narrow, reusable skills
- you should be able to inspect and remix the behavior instead of trusting a black box workflow

That makes it more durable than hypey “one prompt to build your SaaS” repos. It is closer to a library of engineering habits than a magic command.

## Layered architecture dissection

### High-level system shape

This repo has four real layers:

1. discovery layer, where humans and agents find the catalog
2. registry layer, where installable skills are declared
3. execution layer, where each skill encodes behavior and references
4. governance layer, where the repo decides what is public, private, deprecated, or load-bearing

That is a good shape for a prompt-asset repo because it keeps “how to browse”, “what is active”, and “how it behaves” separate.

### Main layers

#### 1. Discovery and product-contract layer

The top-level <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/README.md> does more than sell the project. It maps common coding-agent failure modes to specific skills, which gives the catalog a problem-first entrypoint.

The repo-level <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/CLAUDE.md> is the hidden contract that keeps the catalog coherent. It defines bucket rules, says which skills must appear in the README and plugin manifest, and explicitly excludes `personal/` and `deprecated/` from promotion. That file is doing catalog governance, not just agent flavor text.

#### 2. Registry and packaging layer

The plugin manifest in <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/.claude-plugin/plugin.json> is the machine-readable active set. It only includes the skills intended to be installed into the plugin surface.

The linker script in <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/scripts/link-skills.sh> is tiny but useful. It walks the repo for `SKILL.md` files, symlinks each skill directory into `~/.claude/skills`, and contains a guard against accidentally symlinking back into the repo itself. That is a simple but thoughtful anti-footgun.

#### 3. Skill-execution layer

The real payload lives under <https://github.com/mattpocock/skills/tree/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/skills>.

A few key examples:

- <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/skills/engineering/grill-with-docs/SKILL.md> pushes the agent into an interview loop, but ties it to `CONTEXT.md` and ADR files so the conversation sharpens the domain model instead of free-associating.
- <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/skills/engineering/tdd/SKILL.md> encodes a strong vertical-slice TDD doctrine and explicitly rejects the “write all tests first” anti-pattern.
- <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/skills/engineering/setup-matt-pocock-skills/SKILL.md> is effectively a bootstrap skill that teaches the rest of the engineering set where backlog, labels, and domain docs live.
- <https://github.com/mattpocock/skills/tree/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/skills/engineering/improve-codebase-architecture> shows the pattern of bundling extra reference docs beside `SKILL.md`, which turns a skill from a single prompt into a mini knowledge pack.

This is the repo’s smartest structural choice. Skills are directories, not just files, so a skill can carry supporting doctrine next to the invocation entrypoint.

#### 4. Governance and evolution layer

The ADR at <https://github.com/mattpocock/skills/blob/5fed805a92ddf70dedf1f32c6aadb2a08aaf4d9c/docs/adr/0001-explicit-setup-pointer-only-for-hard-dependencies.md> is a good example of what the maintainers are optimizing for. It explains why some engineering skills mention `/setup-matt-pocock-skills` explicitly while others only refer vaguely to domain docs. That is a token-budget and dependency-management decision, not just editorial cleanup.

The `personal/` and `deprecated/` buckets matter too. They give the repo a lifecycle model: promoted skills, private skills, and retired skills can all coexist without pretending they are equally production-worthy.

### Request / data / control flow

The runtime flow is roughly:

1. a user discovers a skill through the top-level README or plugin setup flow
2. the plugin manifest or symlink install exposes the skill directory to the agent runtime
3. the agent opens that directory’s `SKILL.md`
4. the skill may pull adjacent reference files like `ADR-FORMAT.md`, `tests.md`, or `domain.md`
5. the repo-level `CLAUDE.md` rules constrain how the catalog should be maintained over time

So the repo is not “executing” in a software-runtime sense. It is executing by shaping the agent’s control flow and retrieval context.

## Important knobs / configs / extension points

The main extension points are surprisingly clean:

- bucket placement under `skills/` decides lifecycle and intended audience
- `.claude-plugin/plugin.json` decides what is publicly installable
- top-level and per-bucket `README.md` files decide the discovery surface
- each skill directory can carry its own auxiliary docs, effectively extending the skill with local retrieval context
- `CLAUDE.md` defines maintenance invariants, especially which buckets must or must not be mirrored into the public catalog

This means adding a new skill is mostly content architecture, not code.

## Practical questions and answers

**Q: Is this just a prompt library?**  
Not really. The better model is “prompt components plus governance.” A raw prompt library usually has no lifecycle rules, no registry boundary, and no clear separation between active and deprecated assets.

**Q: What is the cleverest idea here?**  
Treating a skill as a directory, not a single file. That allows a prompt entrypoint plus bundled reference material, which is much closer to how real engineering playbooks work.

**Q: Where would this break down?**  
At scale, metadata drift becomes a real risk. The same skill identity is represented in the bucket README, top-level README, plugin manifest, and sometimes repo-level rules. Without automation, those can diverge.

**Q: Is there real architecture here, or just strong prose?**  
There is real architecture. The architecture is in the repo boundaries and information flows, not in executable code volume.

## What is smart

A few things are genuinely sharp:

- small composable skills instead of one mega-framework
- explicit separation of promoted, personal, and deprecated assets
- using repo-level governance docs to keep the catalog coherent
- packaging companion references with skills so deep workflows can stay lightweight at the entrypoint
- making setup a first-class skill instead of assuming every repo has the same backlog and domain-doc structure
- the `link-skills.sh` anti-recursion guard, which shows someone thought about real-world failure modes

## What is flawed or weak

The repo has some obvious weak spots too:

- governance is duplicated across README files, `CLAUDE.md`, and `plugin.json`
- there is not much automation visible for enforcing those invariants
- skill quality depends heavily on editorial discipline and maintainer taste
- some of the value is personality-bound, because the repo encodes one engineer’s philosophy and vocabulary very strongly
- the repo is more portable as inspiration than as a drop-in standard, because every team’s workflow language differs

## What we can learn / steal

The best reusable lesson is not any one skill. It is the packaging model.

Steal this:
- keep agent behaviors small and named around failure modes
- make repo governance explicit
- separate public, private, and deprecated prompts cleanly
- let a skill ship with adjacent reference docs
- add bootstrap/setup skills whenever downstream prompts rely on repo-local conventions

That is a much better pattern than throwing giant prompts at agents and hoping consistency emerges.

## How we could apply it

For our own work, the most useful adaptation would be:

1. define a small house catalog of high-leverage skills
2. give each skill a folder, not just a markdown file
3. add one repo-level governance doc that explains lifecycle and registration rules
4. create setup/bootstrap skills wherever prompts depend on project-local conventions
5. automate drift detection between the public catalog, registry, and actual skill folders

That last step is the missing piece in this repo. A simple validator script would remove a lot of future entropy.

## Bottom line

`mattpocock/skills` deserves the hype more than most trending agent repos. It is not pretending prompts are software in the naïve sense, but it is treating them as maintainable engineering assets with structure, layering, and lifecycle.

The key insight is simple and worth keeping: the best agent systems may look less like autonomous robots and more like well-curated libraries of narrow operational doctrine.
