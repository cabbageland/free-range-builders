# OpenFang

- Repo: `RightNow-AI/openfang`
- URL: https://github.com/RightNow-AI/openfang
- Date: 2026-05-30
- Repo snapshot studied: `main` @ `acf2587e46be174c10200489c9a2d23a39a98aeb`
- Why picked today: It is a hot, very visible AI-agent repo right now, but unlike a lot of agent hype it at least has a real systems-shaped codebase behind the pitch. It is also interesting precisely because the repo contains both serious infrastructure work and a lot of aggressive marketing, which makes it worth dissecting rather than admiring from a distance.

## Executive summary
OpenFang is trying to turn “AI agent framework” into “agent operating system.” In repo terms, that means a large Rust workspace with a kernel, runtime, memory substrate, HTTP API, channel adapters, network transport, curated autonomous agent bundles called Hands, a skill/plugin system, SDKs, and deployment assets.

The good news is that this is not just a landing page with a loop around an LLM. The repo has real architectural intent. The best idea is the split between durable system layers: kernel orchestration, runtime execution, memory, channels, wire protocol, and packaged autonomous capabilities. The weak point is that the product story often outruns the architecture. The codebase is substantial, but it also tries to be everything at once: daemon, agent runtime, dashboard backend, plugin host, browser automation system, messaging bridge, scheduler, memory store, and network node.

My take: this is a serious repo with real implementation weight, but you should trust the source tree more than the README claims.

## What they built
They built a multi-crate Rust workspace for a self-hosted agent platform with:
- a workspace root in [`Cargo.toml`](https://github.com/RightNow-AI/openfang/blob/main/Cargo.toml),
- a CLI and daemon entry surface in [`crates/openfang-cli/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-cli),
- the system assembly layer in [`crates/openfang-kernel/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-kernel),
- the execution/runtime layer in [`crates/openfang-runtime/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-runtime),
- persistence and knowledge storage in [`crates/openfang-memory/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-memory),
- the HTTP/WebSocket daemon API in [`crates/openfang-api/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-api),
- cross-machine networking in [`crates/openfang-wire/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-wire),
- messaging integrations in [`crates/openfang-channels/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-channels),
- bundled “Hands” in [`crates/openfang-hands/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-hands) and prompt/tool skills in [`crates/openfang-skills/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-skills),
- plus SDKs in [`sdk/javascript/`](https://github.com/RightNow-AI/openfang/tree/main/sdk/javascript) and [`sdk/python/`](https://github.com/RightNow-AI/openfang/tree/main/sdk/python).

This is much closer to a small agent platform product than to a library.

## Why it matters
Most agent repos are still one of three things: a chat wrapper, an orchestration DSL, or a demo-heavy prompt bundle. OpenFang is interesting because it instead asks, “what if an agent system looked like an operating substrate?”

That framing matters because it forces repo-level questions that many AI projects dodge:
- where does persistent state live?
- who schedules work?
- how are tools governed?
- how do agents survive restarts?
- how are channels normalized?
- what is the extension model?
- where does auditability sit?

Even if you dislike the branding, those are the right questions.

## Layered architecture dissection
### High-level system shape
At a high level, OpenFang is built like this:
1. a CLI starts or talks to a daemon,
2. the kernel assembles every major subsystem,
3. the runtime executes agents and tools,
4. memory stores state and session data,
5. API and channels move messages in and out,
6. Hands and skills package higher-level behavior on top.

That is a healthier structure than “LLM loop in app.ts plus plugins.”

### Main layers
**1. Workspace and packaging layer**
- [`Cargo.toml`](https://github.com/RightNow-AI/openfang/blob/main/Cargo.toml) defines a 14-crate workspace with shared dependency policy and release profiles.
- The workspace root is doing real systems work: common async runtime, SQLite, Wasmtime, Axum, reqwest, and security primitives are all standardized there.

This matters because the project is big enough that dependency discipline is architecture.

**2. Operator interface layer**
- [`crates/openfang-cli/src/main.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-cli/src/main.rs) is the front door. It exposes init, start, stop, chat, dashboard, models, skills, channels, approvals, cron, sessions, and more.
- The CLI is not just a thin wrapper. It is the local control plane for the daemon-backed system.

This is one of the cleanest signals that the project thinks in terms of “product runtime” instead of “dev library.”

**3. Kernel orchestration layer**
- [`crates/openfang-kernel/src/lib.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-kernel/src/lib.rs) declares the kernel’s concerns: approval, auth, cron, scheduler, event bus, pairing, workflow, heartbeat, background execution, and more.
- [`crates/openfang-kernel/src/kernel.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-kernel/src/kernel.rs) is the real composition root. `OpenFangKernel` owns registry, scheduler, memory, workflows, audit log, metering, MCP connections, browser context, web context, hands, skills, channels, approvals, process manager, and wire networking.

This is both impressive and slightly worrying. Impressive because the system is genuinely integrated. Worrying because the kernel struct is becoming a god object.

**4. Runtime execution layer**
- [`crates/openfang-runtime/src/lib.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-runtime/src/lib.rs) exposes the actual execution surface: agent loop, tool runner, browser, MCP, sandbox, routing, embeddings, TTS, media understanding, web search/fetch, process management, and more.
- [`crates/openfang-runtime/src/agent_loop.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-runtime/src/agent_loop.rs) appears to be the core conversational/action loop.
- [`crates/openfang-runtime/src/tool_runner.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-runtime/src/tool_runner.rs) is where tool policy becomes concrete runtime behavior.
- [`crates/openfang-runtime/src/sandbox.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-runtime/src/sandbox.rs) houses the WASM sandboxing story.

This runtime crate is effectively the agent VM.

**5. Persistence and memory layer**
- [`crates/openfang-memory/src/lib.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-memory/src/lib.rs) lays out the memory model: structured store, semantic store, knowledge graph, session store, and consolidation engine.
- [`crates/openfang-memory/src/substrate.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-memory/src/substrate.rs) is the most important file here. `MemorySubstrate` composes the stores over SQLite and optionally routes semantic memory through an HTTP backend.

This is a strong design move. “Memory” is not hand-wavy context magic here. It is an explicit substrate with multiple store types.

**6. Transport and integration layer**
- [`crates/openfang-api/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-api) is the daemon-facing HTTP/WebSocket surface.
- [`crates/openfang-wire/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-wire) defines peer discovery and cross-machine communication primitives.
- [`crates/openfang-channels/src/lib.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-channels/src/lib.rs) fanouts a large adapter surface: Slack, Telegram, Discord, WhatsApp, Teams, Matrix, email, Reddit, webhook, and many more.

This is where the repo becomes “platform-shaped” instead of “SDK-shaped.”

**7. Capability packaging layer**
- [`crates/openfang-hands/src/lib.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-hands/src/lib.rs) defines Hands as curated autonomous capability packages with manifests, guardrails, requirements, settings, and metrics.
- [`crates/openfang-hands/src/registry.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-hands/src/registry.rs) shows the operational reality: load bundled hands, scan workspace hands, persist active state, and audit manifest reloads.
- [`crates/openfang-skills/src/lib.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-skills/src/lib.rs) defines the skill runtime matrix: Python, WASM, Node, Shell, Builtin, and PromptOnly.
- [`crates/openfang-skills/src/registry.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-skills/src/registry.rs) handles loading, compatibility conversion from OpenClaw-style skill dirs, prompt-injection scanning, and config injection.

This is one of the repo’s strongest sections. The packaging model is opinionated and concrete.

### Request / data / control flow
A simplified control flow looks like this:
1. an operator runs a command through [`crates/openfang-cli/src/main.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-cli/src/main.rs),
2. the CLI either boots or contacts the daemon exposed by [`crates/openfang-api/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-api),
3. the daemon relies on [`crates/openfang-kernel/src/kernel.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-kernel/src/kernel.rs) to assemble system state,
4. the kernel calls into [`crates/openfang-runtime/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-runtime) for agent-loop execution, tool invocation, browser actions, web search, or MCP access,
5. durable state is read from or written to [`crates/openfang-memory/src/substrate.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-memory/src/substrate.rs),
6. inbound and outbound messages traverse [`crates/openfang-channels/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-channels) or [`crates/openfang-wire/`](https://github.com/RightNow-AI/openfang/tree/main/crates/openfang-wire),
7. higher-level autonomous behavior is packaged as Hands or skills and re-enters the same runtime/kernel path.

The key insight is that the system is organized around durable control surfaces, not just prompt templates.

[`crates/openfang-kernel/src/kernel.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-kernel/src/kernel.rs) tells you what OpenFang really is: not just agent chat, but a large integrated runtime with approvals, audit, metering, channels, browser, web, pairing, cron, MCP, process management, and hand registries.

[`crates/openfang-skills/src/registry.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-skills/src/registry.rs) is notable because it is not naïve about prompt bundles. **Channel normalization as a first-class subsystem**
[`crates/openfang-channels/src/lib.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-channels/src/lib.rs) shows how much of this project is really integration plumbing. That is not glamorous, but it is product-real.

## Practical questions and answers
**Is this actually an “operating system”?**
Not literally, of course. But “agent platform runtime with durable subsystems” is fair. The kernel, memory substrate, scheduler, approvals, channels, and process manager justify the metaphor more than most repos do.

**Where is the architectural center of gravity?**
[`crates/openfang-kernel/src/kernel.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-kernel/src/kernel.rs). If you understand that file and the crates it composes, you understand the repo.

**What is the most reusable design idea?**
Treat agent state as a layered substrate, not as one vector store plus chat transcripts. [`crates/openfang-memory/src/substrate.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-memory/src/substrate.rs) is the best “steal this” part.

**What looks most production-relevant?**
The boring parts: approvals, audit, persistence, process management, channel normalization, and provider fallback. Those are the pieces that make a long-running system survivable.

**What looks most brittle?**
The sheer surface area. The repo wants to own a lot of concerns at once, and the kernel already reflects that. A giant integrated control plane is powerful, but it also raises maintenance and testing pressure everywhere.

**Where does marketing get ahead of mechanism?**
Mostly in the README benchmarking and positioning sections in [`README.md`](https://github.com/RightNow-AI/openfang/blob/main/README.md). The source is real; the comparative framing is much less trustworthy.

## What is smart
- The subsystem split across crates is mostly sensible.
- [`crates/openfang-memory/src/substrate.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-memory/src/substrate.rs) shows a mature understanding that memory is plural, not singular.
- [`crates/openfang-hands/src/registry.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-hands/src/registry.rs) persists active hand state and audits manifest loads, which is practical systems thinking.
- [`crates/openfang-skills/src/registry.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-skills/src/registry.rs) taking prompt injection seriously is refreshing.
- [`crates/openfang-runtime/src/browser.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-runtime/src/browser.rs) keeping browser automation and SSRF protections in one native layer is cleaner than glueing three runtimes together.
- The CLI in [`crates/openfang-cli/src/main.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-cli/src/main.rs) exposes operational concerns directly instead of pretending operations do not exist.

## What is flawed or weak
- [`README.md`](https://github.com/RightNow-AI/openfang/blob/main/README.md) is overloaded with adversarial comparison energy and benchmark theater. That makes it harder to trust the pitch, even when the code is substantial.
- [`crates/openfang-kernel/src/kernel.rs`](https://github.com/RightNow-AI/openfang/blob/main/crates/openfang-kernel/src/kernel.rs) is dangerously close to a “everything lives here” super-struct.
- The repo surface is so broad that quality assurance becomes a product in itself: channels, browser, peer networking, skills, hands, memory, API, cron, SDKs, and desktop all at once.
- The “one binary solves it all” posture is attractive, but the architectural cost is concentrated complexity and a very high regression blast radius.
- Some of the agent/persona inventory in [`agents/`](https://github.com/RightNow-AI/openfang/tree/main/agents) feels more like distribution packaging than core system advantage.

## What we can learn / steal
- Build AI systems around durable operational boundaries, not just model calls.
- Separate orchestration from execution from memory from transport.
- Package autonomy as explicit capability bundles with settings, requirements, and metrics.
- Treat skill loading as a security problem, not just a DX problem.
- Keep browser/network protections near the low-level mechanism, not as a late middleware afterthought.

## How we could apply it
If we borrowed from OpenFang, I would copy the shape, not the whole product ambition:
- a clear kernel/composition root,
- a multi-store memory substrate,
- a strict tool and approval layer,
- packaged autonomous jobs with explicit requirements,
- and native, security-aware browser/web primitives.

I would not copy the breadth-first product sprawl. I would pick one or two operational surfaces and make them excellent before trying to own 40 channels and every agent mode.

## Bottom line
OpenFang is one of the more substantial agent-infra repos in the current wave because it actually has a systems architecture, not just a story. The most important thing to understand is that its real innovation is not “autonomous agents” as such, but the attempt to give them durable operating substrate: kernel, runtime, memory, channels, approvals, and packaged capabilities.

The repo is worth studying, but with a skeptical eye. Admire the substrate. Discount the benchmark theater. Steal the architecture, not the swagger.
