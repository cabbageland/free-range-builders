# Semble

- Date: 2026-05-20

- Why picked today: It is one of the hotter agent-tooling repos circulating right now, it makes a concrete performance claim that is easy to inspect skeptically in source, and it is a more interesting engineering artifact than yet another generic agent shell.

## Executive summary
Semble is not trying to be a full code-intelligence graph or a heavyweight code-RAG platform. It is trying to be a very fast retrieval primitive for agents: walk a repo, chunk code, build a sparse BM25 index plus a dense vector index, then answer natural-language code queries with a tiny, targeted set of snippets.

The design taste is good. Instead of selling mystery-model magic, it keeps the pipeline short and legible: file walking in [`src/semble/index/file_walker.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/index/file_walker.py), syntax-aware chunking in [`src/semble/chunking/`](https://github.com/MinishLab/semble/tree/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/chunking), hybrid retrieval in [`src/semble/search.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/search.py), and thin delivery surfaces in [`src/semble/cli.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/cli.py) and [`src/semble/mcp.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/mcp.py).

The most important insight is that Semble wins by being a **fast snippet selector**, not by becoming a broad agent runtime. That narrower ambition is probably why the repo feels coherent.

## What they built
They built a local code-search engine for agents with four main pieces:

1. a **repo indexing pipeline** that walks files, respects ignore patterns, detects languages, and skips oversized files via [`src/semble/index/create.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/index/create.py) and [`src/semble/index/file_walker.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/index/file_walker.py)
2. a **chunking layer** that prefers syntax-aware chunk boundaries but falls back to line chunking in [`src/semble/chunking/chunking.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/chunking/chunking.py)
3. a **hybrid retrieval layer** that combines dense similarity, BM25, reciprocal-rank fusion, and custom reranking in [`src/semble/search.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/search.py) and [`src/semble/ranking/`](https://github.com/MinishLab/semble/tree/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/ranking)
4. agent-facing **delivery surfaces** as both an MCP server and a CLI in [`src/semble/mcp.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/mcp.py) and [`src/semble/cli.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/cli.py)

## Why it matters
Most agent code exploration is still wasteful. The model spends money discovering where to look before it starts understanding anything.

Semble attacks that waste directly. It tries to replace grep-plus-read thrash with a cheap precomputed retrieval surface. That is useful even if you do not buy every benchmark number in [`benchmarks/`](https://github.com/MinishLab/semble/tree/ef7d3fe78086faf75bde402495bf8d4c3f130093/benchmarks), because the core thesis is directionally right: agent performance often improves more from **better retrieval boundaries** than from more model horsepower.

## Layered architecture dissection
### High-level system shape
The system is basically:

1. collect files
2. split them into retrievable chunks
3. build sparse and dense indexes over those chunks
4. blend and rerank results
5. expose search through CLI or MCP

There is no giant orchestration layer. That is a strength here.

### Main layers
**1. File selection and ignore layer**

[`src/semble/index/file_walker.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/index/file_walker.py) is more important than it looks. It bakes in default ignored directories like `.git`, `node_modules`, virtualenvs, caches, and `.semble/`, while also honoring both `.gitignore` and `.sembleignore`. That means retrieval quality starts with disciplined corpus hygiene, not just model choice.

**2. Chunking layer**

[`src/semble/chunking/chunking.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/chunking/chunking.py) sets a target chunk size around 1500 characters, prefers syntax-aware chunk boundaries through [`src/semble/chunking/core.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/chunking/core.py), and falls back to line-based chunking when parsing support is unavailable. That is the right compromise. It keeps the system fast and robust without pretending every language deserves deep semantic parsing.

**3. Index construction layer**

[`src/semble/index/create.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/index/create.py) orchestrates index creation. It walks eligible files, skips files above 1 MB, reads text defensively, builds chunk objects, embeds them with [`src/semble/index/dense.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/index/dense.py), and builds a BM25 index after enriching tokens for sparse retrieval. This is a clean, small pipeline.

**4. Retrieval and reranking layer**

[`src/semble/search.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/search.py) is the heart. It runs semantic search and BM25 independently, converts both to reciprocal-rank-fusion style scores, unions candidates, then reranks. The reranking logic in [`src/semble/ranking/boosting.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/ranking/boosting.py), [`src/semble/ranking/penalties.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/ranking/penalties.py), and [`src/semble/ranking/weighting.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/ranking/weighting.py) is where much of the practical quality likely comes from.

**5. Delivery layer**

[`src/semble/cli.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/cli.py) exposes `search`, `find-related`, `init`, and `savings`. [`src/semble/mcp.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/mcp.py) wraps the same core index in a FastMCP server, adds an LRU-style cache of indexed repos, permits remote git URLs, and re-indexes local paths on file change using `watchfiles`.

### Request / data / control flow
Typical local flow:

1. CLI or MCP entrypoint receives a query in [`src/semble/cli.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/cli.py) or [`src/semble/mcp.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/mcp.py).
2. A [`SembleIndex`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/index/index.py) instance is built from a local path or a shallow git clone.
3. [`create_index_from_path`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/index/create.py) walks files, chunks source, computes embeddings, and creates BM25 state.
4. [`search`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/search.py) executes dense and sparse retrieval, fuses ranks, and reranks candidates.
5. [`src/semble/utils.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/utils.py) formats compact results for the agent.
6. If the user asks for similarity from a known location, [`SembleIndex.find_related`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/index/index.py) reuses semantic search over chunk content only.

That is a very economical control flow. Most of the repo exists to make those six steps fast and dependable.

## Practical questions and answers
**Is this semantic search only?**
No. The repo is explicitly hybrid. Dense retrieval gets combined with BM25 in [`src/semble/search.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/search.py), then reranked.

**What is the best engineering choice here?**
Not overcommitting to a giant architecture. The product stays legible because the core is just chunking plus dual indexing plus reranking.

**Where is most of the quality probably coming from?**
I suspect the reranking heuristics matter more than the README admits. Symbol-definition boosts and multi-chunk file coherence in [`src/semble/ranking/boosting.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/ranking/boosting.py) look like the kind of boring details that make search feel much smarter.

**What is the repo not solving?**
It does not build a durable semantic graph of inter-symbol relationships, call graphs, or architecture boundaries. It is retrieval, not repository understanding in the richer structural sense.

**What breaks first at scale?**
Large monorepos with generated code, vendored code, or important cross-file flows that require deeper graph reasoning may expose the limits of chunk-level retrieval.

## What is smart
- The architecture is disciplined and intentionally small.
- They use syntax-aware chunking where available, but keep a robust fallback.
- They do hybrid retrieval rather than ideology, sparse or dense alone.
- The MCP layer in [`src/semble/mcp.py`](https://github.com/MinishLab/semble/blob/ef7d3fe78086faf75bde402495bf8d4c3f130093/src/semble/mcp.py) is thin, which keeps product latency close to core-engine latency.
- `.sembleignore` is a good local-control detail. Retrieval quality depends heavily on corpus selection.

## What is flawed or weak
- The benchmark story in [`benchmarks/`](https://github.com/MinishLab/semble/tree/ef7d3fe78086faf75bde402495bf8d4c3f130093/benchmarks) is interesting, but still self-reported. It should not be treated as settled truth.
- The system is chunk-centric, which means cross-file architecture questions may still require a stronger second-stage explorer.
- Hardcoded chunk sizing and file-size limits are sensible defaults, but they are opinionated enough that some repos will want more tuning.
- Remote git support is convenient, but shallow clone plus in-memory indexing means this is optimized for fast lookup sessions, not durable multi-repo knowledge accumulation.
- The product claim can sound broader than the implementation. Semble helps agents find code quickly; it does not by itself make them understand systems deeply.

## What we can learn / steal
- Keep the core retrieval pipeline painfully short.
- Put real thought into chunk boundaries and ignore semantics before touching fancy model work.
- Blend sparse and dense retrieval, then spend effort on reranking details.
- Treat MCP as a delivery shell around a sharp core primitive, not the product itself.
- Build small tools that do one thing very well for agents, instead of sprawling into full-stack agent platforms too early.

## How we could apply it
If we wanted an internal repo assistant, the best thing to steal is the decomposition:

1. aggressively curate the corpus
2. chunk source with structural awareness
3. combine lexical and semantic retrieval
4. add cheap reranking rules for symbol definitions and file coherence
5. return compact snippets first, not full files

That would be especially useful for sub-agents, quick code archaeology, and any workflow where discovery cost dominates reasoning cost.

## Bottom line
Semble is impressive because it stays narrow. It is a retrieval engine for agents, not a kitchen-sink AI coding platform.

My main takeaway: the repo's leverage comes less from exotic ML and more from smart retrieval plumbing, especially chunking discipline, hybrid ranking, and lightweight heuristics layered over a very small core.
