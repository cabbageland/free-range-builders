## Executive summary
Cua is a serious attempt to own the whole computer-use agent stack instead of only shipping one model wrapper. The repo combines three hard things that are usually split across companies: an agent SDK, cross-OS sandbox/runtime control, and benchmark/eval infrastructure. The most interesting part is not the prompting layer. It is the infrastructure layer underneath, especially the way the repo decomposes computer use into reusable transport, runtime, driver, and benchmark subsystems.

## What they built
They built a monorepo for computer-use agents with several product lines under one roof:

- a unified Python-facing sandbox/agent SDK in [`libs/python`](https://github.com/trycua/cua/tree/main/libs/python)
- a benchmark and RL/eval system in [`libs/cua-bench`](https://github.com/trycua/cua/tree/main/libs/cua-bench)
- a native macOS background driver in [`libs/cua-driver`](https://github.com/trycua/cua/tree/main/libs/cua-driver)
- a local VM manager for Apple Silicon in [`libs/lume`](https://github.com/trycua/cua/tree/main/libs/lume)
- a desktop-oriented multi-agent sandbox launcher in [`libs/cuabot`](https://github.com/trycua/cua/tree/main/libs/cuabot)
- docs, examples, notebooks, blog posts, and test infrastructure around all of it

## Why it matters
Most "computer use" repos are thin wrappers around screenshots plus click/type tools. Cua is trying to turn the category into infrastructure. That is strategically smarter. If they become the default runtime/sandbox/eval substrate, they can swap models above it without losing the platform position.

## Layered architecture dissection
### High-level system shape
The system looks like this:

1. **Runtime layer** provisions a computer target, local or cloud.
2. **Transport/interface layer** exposes screen, shell, keyboard, mouse, files, and app controls.
3. **Agent layer** runs model loops and tool orchestration against those interfaces.
4. **Driver layer** handles hard native interaction problems, especially on macOS.
5. **Benchmark layer** evaluates agents in repeatable environments.
6. **Operator/developer layer** wraps all of it in CLIs, docs, notebooks, and desktop UX.

### Main layers
**1. Sandbox and runtime abstraction**

The center of gravity is [`libs/python/cua-sandbox/cua_sandbox`](https://github.com/trycua/cua/tree/main/libs/python/cua-sandbox/cua_sandbox). Important files include:

- [`sandbox.py`](https://github.com/trycua/cua/blob/main/libs/python/cua-sandbox/cua_sandbox/sandbox.py), the main lifecycle abstraction
- [`image.py`](https://github.com/trycua/cua/blob/main/libs/python/cua-sandbox/cua_sandbox/image.py), which defines image selection semantics
- [`runtime`](https://github.com/trycua/cua/tree/main/libs/python/cua-sandbox/cua_sandbox/runtime), which fans out into Docker, QEMU, Lume, Tart, Android emulator, and compatibility helpers
- [`transport`](https://github.com/trycua/cua/tree/main/libs/python/cua-sandbox/cua_sandbox/transport), which carries HTTP, SSH, VNC, cloud, ADB, QMP, and websocket transport implementations
- [`interfaces`](https://github.com/trycua/cua/tree/main/libs/python/cua-sandbox/cua_sandbox/interfaces), which normalize actions like mouse, keyboard, screen, shell, clipboard, apps, and windows

This is the most reusable part of the repo. It turns "computer use" from a one-off demo into an execution substrate.

**2. Agent loop layer**

The agent package in [`libs/python/agent/cua_agent`](https://github.com/trycua/cua/tree/main/libs/python/agent/cua_agent) contains:

- model-specific loops in [`loops`](https://github.com/trycua/cua/tree/main/libs/python/agent/cua_agent/loops)
- model/provider adapters in [`adapters`](https://github.com/trycua/cua/tree/main/libs/python/agent/cua_agent/adapters)
- callbacks for budgets, telemetry, logging, PII, trajectory saving, and operator validation in [`callbacks`](https://github.com/trycua/cua/tree/main/libs/python/agent/cua_agent/callbacks)
- computer bindings in [`computers`](https://github.com/trycua/cua/tree/main/libs/python/agent/cua_agent/computers)
- top-level orchestration in [`agent.py`](https://github.com/trycua/cua/blob/main/libs/python/agent/cua_agent/agent.py)

The notable choice here is that the repo treats model loops as pluggable leaves rather than the core product. That keeps the architecture resilient if frontier model preferences change.

**3. Computer server / execution bridge**

[`libs/python/computer-server/computer_server`](https://github.com/trycua/cua/tree/main/libs/python/computer-server/computer_server) looks like the bridge between abstract agent actions and concrete environment operations. Key parts:

- [`server.py`](https://github.com/trycua/cua/blob/main/libs/python/computer-server/computer_server/server.py)
- [`mcp_server.py`](https://github.com/trycua/cua/blob/main/libs/python/computer-server/computer_server/mcp_server.py)
- [`handlers`](https://github.com/trycua/cua/tree/main/libs/python/computer-server/computer_server/handlers) for Linux, macOS, Windows, Android, VNC, and generic backends
- [`diorama`](https://github.com/trycua/cua/tree/main/libs/python/computer-server/computer_server/diorama), which appears to provide structured visual/state handling rather than raw screenshot-only operation

This layer matters because it is where portability usually dies. They are clearly investing in backend-specific handlers instead of pretending every OS behaves the same.

**4. Native macOS background driver**

[`libs/cua-driver`](https://github.com/trycua/cua/tree/main/libs/cua-driver) is one of the repo's sharpest pieces. The core is split into:

Files worth studying:

- [`Focus/FocusGuard.swift`](https://github.com/trycua/cua/blob/main/libs/cua-driver/Sources/CuaDriverCore/Focus/FocusGuard.swift)
- [`Input/MouseInput.swift`](https://github.com/trycua/cua/blob/main/libs/cua-driver/Sources/CuaDriverCore/Input/MouseInput.swift)

The background-driver angle is real differentiation. "Computer use without stealing focus" is much harder than ordinary browser automation.

**5. Benchmark and training layer**

[`libs/cua-bench`](https://github.com/trycua/cua/tree/main/libs/cua-bench) is not a side demo. It is a parallel product line.

Key paths:

- [`cua_bench/agents`](https://github.com/trycua/cua/tree/main/libs/cua-bench/cua_bench/agents)
- [`cua_bench/runner`](https://github.com/trycua/cua/tree/main/libs/cua-bench/cua_bench/runner)
- [`cua_bench/workers`](https://github.com/trycua/cua/tree/main/libs/cua-bench/cua_bench/workers)
- [`cua_bench/telemetry`](https://github.com/trycua/cua/tree/main/libs/cua-bench/cua_bench/telemetry)
- [`datasets`](https://github.com/trycua/cua/tree/main/libs/cua-bench/datasets)
- [`tasks`](https://github.com/trycua/cua/tree/main/libs/cua-bench/tasks)

This means they want to control not just execution, but evaluation and trajectory generation too.

**6. Local virtualization and operator UX**

The local infra side is split between:

- [`libs/lume`](https://github.com/trycua/cua/tree/main/libs/lume), a Swift VM manager on Apple Virtualization Framework
- [`libs/lumier`](https://github.com/trycua/cua/tree/main/libs/lumier), a Docker-like wrapper around that VM substrate
- [`libs/cuabot`](https://github.com/trycua/cua/tree/main/libs/cuabot), which packages sandbox UX for humans and other agents

That combination is the "distribution" layer. It helps them turn infra primitives into something agents and operators can actually adopt.

### Request / data / control flow
A plausible mainline flow looks like:

1. A developer creates a sandbox via [`Sandbox`](https://github.com/trycua/cua/blob/main/libs/python/cua-sandbox/cua_sandbox/sandbox.py).
2. The sandbox selects a runtime backend via the [`runtime`](https://github.com/trycua/cua/tree/main/libs/python/cua-sandbox/cua_sandbox/runtime) package and transport plumbing via [`transport`](https://github.com/trycua/cua/tree/main/libs/python/cua-sandbox/cua_sandbox/transport).
3. An agent loop from [`cua_agent/loops`](https://github.com/trycua/cua/tree/main/libs/python/agent/cua_agent/loops) consumes screenshots/state and emits actions.
4. Those actions route through computer/server handlers in [`computer_server/handlers`](https://github.com/trycua/cua/tree/main/libs/python/computer-server/computer_server/handlers).
5. On macOS native workflows, [`cua-driver`](https://github.com/trycua/cua/tree/main/libs/cua-driver) can provide a deeper interaction primitive set.
6. Benchmarks in [`libs/cua-bench`](https://github.com/trycua/cua/tree/main/libs/cua-bench) reuse the same substrate to run repeatable tasks and capture trajectories.

## Practical questions and answers
**Is this mainly a model repo?**
No. The durable asset is the execution substrate and eval stack, not any one loop implementation.

**What is the strongest part?**
The decomposition into runtime, transport, interface, agent, and benchmark layers. That is a much more transferable architecture than a single vertically tangled app.

**What is most differentiated?**
[`cua-driver`](https://github.com/trycua/cua/tree/main/libs/cua-driver) plus [`lume`](https://github.com/trycua/cua/tree/main/libs/lume). A native background macOS driver paired with local virtualization is a harder moat than "we support tool calling".

**Where could this fail in production?**
Cross-platform surface area. This repo spans Python, TypeScript, Swift, desktop UX, OS automation, transports, and benchmarks. That breadth is powerful, but every OS/backend combination expands the maintenance blast radius.

**What looks slightly over-ambitious?**
The repo tries to be SDK, CLI, driver, benchmark suite, VM manager, playground, docs platform, and desktop wrapper at once. That can compound complexity faster than usage compounds.

## What is smart
- They are building infrastructure below the model layer.
- They separate runtime concerns from agent policy concerns.
- They appear to treat benchmarks and trajectories as first-class, not afterthoughts.
- The repo is polyglot for a reason: Swift where native macOS performance/control matter, Python where agent orchestration matters, TypeScript where distribution and developer tooling benefit.
- The monorepo shape makes sense because the products share primitives.

## What is flawed or weak
- The README is polished, but the repo surface is sprawling enough that onboarding likely feels heavier than the homepage implies.
- Some package naming and product boundaries are clear only after source inspection. A newcomer has to learn Cua, Cua Driver, Cuabot, Lume, Lumier, and Cua Bench together.
- Cross-language monorepos are powerful but costly. Release discipline, version compatibility, and test coverage have to stay excellent or entropy wins.
- This kind of system can hide substantial operational complexity behind a clean SDK veneer.

## What we can learn / steal
- Treat computer use as infrastructure layers, not just a model prompt loop.
- Make transports and runtimes swappable.
- Put benchmark/eval infrastructure in the same architectural conversation as product APIs.
- Use native code only where it actually buys control or performance, instead of forcing everything through one language.
- Expose sharp internals through MCP/tool boundaries so external agents can compose with them.

## How we could apply it
If we were building our own agent platform, the best idea to steal is the split between:

- sandbox lifecycle
- runtime backend
- transport protocol
- tool/action interface
- agent loop
- eval harness

That separation makes it easier to test components independently and replace weak models or runtimes without rewriting the whole system.

A second good steal is the insistence on first-class trajectory capture and benchmark tasks. That gives you a path from demos to training and regression testing.

## Bottom line
Cua is interesting because it is not pretending computer use is just screenshots plus clicking. It is trying to become the substrate for the category. The most important insight is that the repo's moat lives in the infrastructure seams, especially runtime abstraction plus native control plus eval plumbing, not in the top-layer agent loop itself.
