# GBrain

- Repo: `garrytan/gbrain`
- URL: <https://github.com/garrytan/gbrain>
- Date: 2026-04-27
- Repo snapshot studied: `master` @ `891c28b582967154f91016a9d2b9a085f05d56e5`
- Why picked today: It is one of the hottest agent-infra repos right now, and unlike a lot of “AI memory” projects, it is not just a wrapper. It is trying to turn an agent’s memory, retrieval, enrichment, scheduling, and background work into a real operating substrate.

## Executive summary

GBrain is Garry Tan’s opinionated personal-agent knowledge runtime. At the product layer it looks like “memory for your AI agent,” but the repo is more ambitious than that. It combines a markdown-native knowledge base, a hybrid search engine, an auto-wiring graph layer, a library of executable skills, and a durable background job system for agent work.

The strongest read is: GBrain is trying to package the internal operating patterns of a serious always-on assistant, especially one running on OpenClaw or Hermes, into a reusable stack. It is not only storing notes. It is enforcing an agent loop: ingest signals, check the brain first, enrich entities, write structured pages back, auto-link them, and schedule recurring upkeep.

The repo is strongest when it acts like infrastructure, not vibes. The engine abstraction is real, the search layer is explicit, the graph extraction is concrete, and the skill files are treated as executable operating doctrine. The weakness is scope creep. This is simultaneously a CLI, personal wiki indexer, graph layer, MCP server, skillpack runtime, background job control plane, and agent-install recipe. That breadth is powerful, but it makes the repo feel more like a proto-agent OS than a focused tool.

## What GBrain actually is

GBrain is a TypeScript/Bun codebase plus a skillpack that turns a folder of markdown, plus external signals, into an agent-usable knowledge system:

- CLI and runtime entrypoint in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/cli.ts>
- pluggable engine contract in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/core/engine.ts>
- PGLite local engine in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/core/pglite-engine.ts>
- Postgres engine in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/core/postgres-engine.ts>
- import pipeline in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/commands/import.ts>
- link and code-reference extraction in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/core/link-extraction.ts>
- MCP server wrapper in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/commands/serve.ts>
- MCP tool-definition builder in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/mcp/tool-defs.ts>
- bundled agent skills in <https://github.com/garrytan/gbrain/tree/891c28b582967154f91016a9d2b9a085f05d56e5/skills>
- repo-level product contract in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/README.md>

At a practical level, GBrain is doing six jobs at once:

1. indexing markdown and repo content into a queryable knowledge store
2. serving hybrid keyword/vector retrieval over that store
3. auto-building a typed knowledge graph from page writes and references
4. providing agent-facing skills for lookup, enrichment, ingestion, and maintenance
5. exposing the system as an MCP server and CLI
6. orchestrating background jobs and subagents through its Minions layer

## Why it matters

A lot of “agent memory” repos hand-wave the hard part. They store chunks, slap embeddings on them, and call it done. GBrain is more serious about the surrounding operational loop.

The key idea is that memory is not just retrieval. For a real assistant, memory needs:

- ingestion paths
- source attribution
- entity pages
- structured timelines
- graph edges
- recurring maintenance
- background jobs
- a doctrine for when the agent should consult memory before doing anything else

GBrain has opinions on all of those.

## Layered architecture dissection

### High-level system shape

GBrain is best understood as five stacked layers:

1. *storage and engine layer* for PGLite and Postgres-backed persistence
2. *import/search/graph layer* for turning files into retrievable knowledge
3. *agent contract layer* for operations exposed through CLI and MCP
4. *skill layer* for executable workflows and behavior policy
5. *orchestration layer* for scheduled jobs, background tasks, and subagents

That stack is more complete than most repos in this category.

### 1. Storage and engine layer

The core abstraction is `BrainEngine` in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/core/engine.ts>.

This file is important because it shows GBrain is not just a pile of scripts. It has a real interface for:

- page CRUD
- keyword and vector search
- chunk management
- typed links
- timeline entries
- raw data
- transactions
- reserved connections

That interface gets two main concrete implementations:

- local zero-config PGLite in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/core/pglite-engine.ts>
- fuller Postgres implementation in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/core/postgres-engine.ts>

The smart product move is dual-mode deployment. PGLite makes “brain ready in 2 seconds” plausible for solo use. Postgres makes larger multi-worker and durable orchestration features possible.

This is also where the repo feels opinionated in a useful way. It does not pretend all backends are equal. The code comments are full of notes about locking, pool behavior, advisory locks, session defaults, and multi-source correctness. That is infra thinking, not wrapper thinking.

### 2. Import, parsing, and retrieval layer

The infrastructure diagram in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/docs/architecture/infra-layer.md> lays out the pipeline clearly:

- resolve files
- parse markdown and frontmatter
- hash for idempotency
- chunk content
- embed chunks
- store pages and chunks transactionally
- expose hybrid search immediately

The import command in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/commands/import.ts> is not glamorous, but it matters. It handles checkpointing, parallel import for Postgres, progress, per-file failures, and resumability. That is what a real “brain” needs if it is going to ingest thousands of files instead of staying a demo.

On search, the repo’s infra doc is clearer than the marketing. It uses hybrid retrieval with Reciprocal Rank Fusion over keyword and vector results, plus dedup and diversity rules, documented in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/docs/architecture/infra-layer.md>. That is more grounded than the usual “RAG but better” pitch.

### 3. Graph and auto-link layer

The graph layer is one of the repo’s best concrete ideas.

The extraction code in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/core/link-extraction.ts> parses:

- markdown entity links
- Obsidian-style wikilinks
- source-qualified links
- code references

The README and migration docs make the higher-level claim: every page write can automatically extract entity references and typed relationships like `works_at`, `invested_in`, `founded`, `attended`, and `advises`, then fold that back into retrieval.

That matters because most memory systems stop at chunk retrieval. GBrain is trying to make structural relationships first-class. The reported win is not subtle either: better precision and recall because graph structure and backlinks help lift the right page.

The repo is strongest here when it stays deterministic. The graph is not emerging from model magic. It is coming from write-time extraction rules and database updates.

### 4. Agent contract layer: CLI plus MCP

GBrain exposes itself both as a CLI and as an MCP server.

The CLI is the operator and local-agent path. The MCP path is for external agent clients. The serve command in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/commands/serve.ts> is thin because the heavy lifting is in the operations layer and the MCP server.

The trust boundary is called out explicitly in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/AGENTS.md>: local CLI callers are trusted differently from remote MCP callers. That is a good sign. It means the repo is trying to treat “agent can call tools” as a security problem, not only a convenience feature.

The MCP tool-definition builder in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/src/mcp/tool-defs.ts> is simple, but conceptually important. It turns internal operations into an agent-visible contract.

### 5. Skill layer: the real product surface

This repo keeps saying “skill files are code,” and in practice that is not just branding.

The skillpack in <https://github.com/garrytan/gbrain/tree/891c28b582967154f91016a9d2b9a085f05d56e5/skills> is where the product’s intended behavior really lives. A few key files:

- brain-first lookup and read/write doctrine in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/skills/brain-ops/SKILL.md>
- search and synthesis behavior in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/skills/query/SKILL.md>
- person/company enrichment policy in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/skills/enrich/SKILL.md>
- ambient signal capture in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/skills/signal-detector/SKILL.md>
- durable background job routing in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/skills/minion-orchestrator/SKILL.md>

The important insight is that GBrain is not only a database. It is a behavior spec for agents.

For example, the “brain-first lookup” doctrine is spelled out in both the guide at <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/docs/guides/brain-first-lookup.md> and the brain-ops skill. Before using external APIs, the agent is supposed to search the brain first, then query semantically, then fetch by slug, and only then leave the system. That is a strong product opinion, and a good one.

### 6. Orchestration and Minions layer

This is where GBrain starts looking like an agent control plane.

The design memo in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/docs/designs/MINIONS_AGENT_ORCHESTRATION.md> is explicit: Minions is supposed to become a universal agent orchestration protocol, not just an internal queue.

The corresponding skill in <https://github.com/garrytan/gbrain/blob/891c28b582967154f91016a9d2b9a085f05d56e5/skills/minion-orchestrator/SKILL.md> frames two lanes:

- deterministic shell jobs
- LLM subagent jobs

That matters because it expands GBrain from knowledge storage into durable agent work management. Once the same system holds the knowledge base and the background work queue, the agent stack becomes much more self-contained.

This is ambitious, and maybe too ambitious, but it is not fake ambition. There is a coherent architecture behind it.

## How the system is meant to work in practice

The README and skills together imply this loop:

1. a signal arrives, message, meeting, link, email, tweet, repo, note
2. `signal-detector` captures ideas and entities in parallel
3. `brain-ops` checks the brain first before any external lookup
4. `query` or related tools answer from existing context when possible
5. `enrich` updates or creates person/company pages when needed
6. `put_page` and auto-link extraction wire new graph edges
7. recurring jobs and Minions maintain the system over time

That loop is arguably the real product. The database, search, and graph only matter because they support that operating rhythm.

## What is smart

- It treats agent memory as an operational system, not just embeddings.
- The engine abstraction is real and gives it a credible local-to-serious deployment path.
- The graph layer is deterministic and wired into page writes, which is much better than “maybe the LLM inferred a relationship.”
- The “brain-first lookup” doctrine is exactly the kind of opinionated rule agent systems need.
- The skill files are not fluff. They encode behavior, quality bar, routing, and doctrine in a form agents can read directly.
- The Minions direction is conceptually strong if they can keep it from collapsing under scope.

## What feels weak or risky

- The product boundary is fuzzy. Is this a personal knowledge base, an agent installer, an MCP server, a workflow engine, or a job queue? The answer is “yes,” which is exciting but dangerous.
- The repo leans heavily on docs and skill files for coherence. That can be a strength, but it also means behavior is spread across code and markdown contracts.
- There is a lot of ambition around enrichment, scheduling, orchestration, and “universal protocol” language that could outrun the maintainability of the codebase.
- The README makes some very strong benchmark and production-scale claims, but the repo still feels like a fast-moving founder project more than a stabilized platform.
- There is a risk of overfitting to Garry’s own OpenClaw/Hermes operating style. That may be fine, but it means some abstractions might generalize less well than claimed.

## What we can learn / steal

- Treat memory as a loop, not a storage primitive.
- Make “brain first” a hard behavioral rule instead of leaving retrieval discipline to chance.
- Keep the graph layer deterministic and close to writes.
- Encode agent workflows in readable skill files or playbooks, not only in hidden prompt text.
- Separate local easy mode from serious deployment mode without forking the product into two incompatible things.
- If you want long-lived agent systems, background work orchestration eventually wants to live near the knowledge layer.

## What I think GBrain is, in one sentence

GBrain is Garry Tan’s attempt to turn the private operating system of a serious always-on AI assistant, memory, graph, skills, and durable work orchestration included, into a reusable open-source substrate.

## Bottom line

GBrain is not just “memory for agents.” It is a fairly opinionated agent runtime built around a knowledge brain.

The repo’s best insight is that useful agent memory needs structure, doctrine, and ongoing maintenance, not only chunks and vectors. The repo’s main risk is that it wants to be too many layers of the stack at once.

Still, this is one of the more interesting agent-infra repos out right now because it is actually trying to wire the whole loop together.
