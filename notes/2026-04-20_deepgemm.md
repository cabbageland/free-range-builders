# DeepGEMM

- Repo: `deepseek-ai/DeepGEMM`
- URL: https://github.com/deepseek-ai/DeepGEMM
- Date: 2026-04-20
- Repo snapshot studied: `main` @ `7f2a703ed51ac1f7af07f5e1453b2d3267d37d50`
- Why picked today: It is hot on GitHub today, AI-relevant in a real systems way, and interesting because it is not another model wrapper. It is a compact GPU kernel library that tries to make cutting-edge tensor-core work readable enough to study.

## Executive summary

DeepGEMM is a CUDA kernel library for the compute primitives that matter in modern LLM systems: FP8 and FP4 GEMMs, grouped MoE kernels, MQA scoring kernels, HyperConnection kernels, layout transforms, and a fused Mega MoE path. The central architectural bet is that they can keep the public surface small while pushing most of the complexity into a runtime JIT compiler plus a compact set of architecture-specific kernel implementations.

The smart part is the repo shape. The Python package is intentionally thin. The real center of gravity is a C++/CUDA core that exposes a few clean pybind entry points, then dynamically compiles the exact kernels it needs and caches them. That makes the project feel less like a giant template jungle and more like a small kernel operating system for DeepSeek-style workloads.

The weakness is that this is still niche, hardware-opinionated software. It is tightly optimized for SM90 and SM100 class NVIDIA GPUs, and some of the ergonomics are “kernel engineer clean,” not “general library easy.” But as a repo to learn from, it is excellent.

## What they built

They built a unified tensor-kernel library for large-model inference and training primitives, with Python bindings and runtime compilation.

Concretely, the repo provides:

- dense GEMM entry points for FP8, FP4, and BF16 via [`csrc/apis/gemm.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/gemm.hpp)
- attention/indexer kernels via [`csrc/apis/attention.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/attention.hpp)
- layout and scaling-factor transformation helpers via [`csrc/apis/layout.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/layout.hpp) and [`deep_gemm/utils/layout.py`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/deep_gemm/utils/layout.py)
- runtime configuration and JIT init hooks via [`csrc/apis/runtime.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/runtime.hpp)
- a fused multi-rank Mega MoE path via [`csrc/apis/mega.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/mega.hpp) and [`deep_gemm/mega/__init__.py`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/deep_gemm/mega/__init__.py)

So this is not just “fast matmul.” It is a small, opinionated kernel stack for the exact patterns LLM systems hit in production.

## Why it matters

A lot of AI infra repos are orchestration or product wrappers around other people’s kernels. DeepGEMM is lower in the stack. It is where performance, memory layout, kernel launch shape, communication overlap, and hardware-specific tradeoffs actually live.

The key thing worth learning is that they aim for **few public APIs, lots of internal specialization**. That is often a better design than exposing every kernel variation as a new product surface.

## Layered architecture dissection

### High-level system shape

The repo has a four-layer shape:

1. Python import and ergonomics layer in [`deep_gemm/__init__.py`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/deep_gemm/__init__.py)
2. pybind registration layer in [`csrc/python_api.cpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/python_api.cpp)
3. operation-family API layer in [`csrc/apis`](https://github.com/deepseek-ai/DeepGEMM/tree/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis)
4. JIT plus kernel implementation layer in [`csrc/jit`](https://github.com/deepseek-ai/DeepGEMM/tree/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/jit) and [`csrc/jit_kernels`](https://github.com/deepseek-ai/DeepGEMM/tree/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/jit_kernels)

That layering is the repo’s best property. It keeps the user-facing package small while still leaving room for serious low-level specialization.

### Main layers

#### 1) Python package layer

[`deep_gemm/__init__.py`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/deep_gemm/__init__.py) is doing three jobs:

- exposing the public API names
- importing optional feature families conditionally
- bootstrapping the C++ runtime with `_C.init(...)`

That file is thin by design. The Python package is mostly a stable import surface, not the main implementation site.

#### 2) Binding layer

[`csrc/python_api.cpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/python_api.cpp) is deliberately tiny. It just registers the operation families: attention, einsum, hyperconnection, gemm, layout, mega, and runtime.

That is a useful design tell. They want one extension module with several coherent subsystems, not many scattered extension modules.

#### 3) Operation-family API layer

The headers under [`csrc/apis`](https://github.com/deepseek-ai/DeepGEMM/tree/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis) are where the project starts feeling productized.

For example, [`csrc/apis/gemm.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/gemm.hpp) does:

- shape and dtype checks
- memory-layout checks
- early returns for trivial cases
- scale-factor layout transformation
- architecture dispatch to SM90 vs SM100 implementations

So the API layer is not just binding glue. It is the domain boundary where user tensors become validated kernel-launch requests.

#### 4) JIT/runtime layer

[`csrc/jit/compiler.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/jit/compiler.hpp) is one of the most instructive files in the repo. The compiler runtime:

- derives a cache key from kernel name, compiler signature, flags, and generated code
- compiles into a temporary directory
- fsyncs artifacts for visibility across distributed filesystems
- atomically renames the compiled directory into cache
- reuses cached runtimes across processes

That fsync plus atomic-directory-rename detail is the kind of thing you only add after living through real cluster pain. It is a very good sign.

#### 5) Heuristics and architecture-specific kernels

The split between [`csrc/jit_kernels/heuristics`](https://github.com/deepseek-ai/DeepGEMM/tree/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/jit_kernels/heuristics) and [`csrc/jit_kernels/impls`](https://github.com/deepseek-ai/DeepGEMM/tree/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/jit_kernels/impls) is another smart choice.

It means kernel selection logic is not completely tangled with kernel bodies. That makes the repo easier to evolve as hardware generations change.

#### 6) Mega MoE subsystem

The Mega MoE path is its own mini-architecture.

[`deep_gemm/mega/__init__.py`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/deep_gemm/mega/__init__.py) handles Python-side symmetric-memory buffer management and weight transforms. [`csrc/apis/mega.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/mega.hpp) defines the packed shared buffer layout and launches the fused kernel. [`tests/test_mega_moe.py`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/tests/test_mega_moe.py) shows the end-to-end baseline-vs-fused story.

This is a good example of a repo where the test file is also an architecture document.

### Request / data / control flow

Typical flow looks like this:

1. Python user calls `deep_gemm.fp8_*`, `bf16_*`, MQA, or Mega MoE API from [`deep_gemm/__init__.py`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/deep_gemm/__init__.py)
2. call lands in the single extension module registered by [`csrc/python_api.cpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/python_api.cpp)
3. API-layer code validates tensor shape, dtype, layout, and scaling-factor format in files like [`csrc/apis/gemm.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/gemm.hpp)
4. runtime helpers from [`csrc/apis/runtime.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/runtime.hpp) and [`csrc/jit`](https://github.com/deepseek-ai/DeepGEMM/tree/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/jit) prepare architecture state, compiler state, and cache access
5. heuristics choose the relevant implementation family under [`csrc/jit_kernels/heuristics`](https://github.com/deepseek-ai/DeepGEMM/tree/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/jit_kernels/heuristics)
6. compiled kernel implementations from [`csrc/jit_kernels/impls`](https://github.com/deepseek-ai/DeepGEMM/tree/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/jit_kernels/impls) execute on the target GPU

The important point is that the public API is stable while codegen and dispatch remain dynamic.

## Important components

### `Compiler` and cache machinery

[`csrc/jit/compiler.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/jit/compiler.hpp) is the component I would steal first. It treats kernel compilation as a production systems problem, not just a dev convenience.

### API-family headers

The API headers, especially [`csrc/apis/gemm.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/gemm.hpp), do the hard work of validating layout assumptions and routing requests cleanly. That keeps the kernel bodies from being cluttered with caller-safety logic.

### Mega MoE shared-buffer design

[`csrc/apis/mega.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/mega.hpp) lays out a big symmetric-memory workspace and slices typed tensor views out of it. That is a very concrete example of “communication and compute fused by memory contract.”

### Runtime knobs

[`csrc/apis/runtime.hpp`](https://github.com/deepseek-ai/DeepGEMM/blob/7f2a703ed51ac1f7af07f5e1453b2d3267d37d50/csrc/apis/runtime.hpp) exposes just enough tuning surface to make experimentation possible without turning the library into a kitchen sink.

## Practical questions and answers

### Why does this repo feel cleaner than many high-performance CUDA repos?

Because they aggressively compress the public surface area. There are relatively few top-level APIs compared with the amount of internal specialization.

### Where is the real architectural trick?

In the JIT layer. They use runtime compilation and caching to avoid baking every specialization into a giant ahead-of-time build and to keep installation lighter.

### What is the most reusable idea here for normal infra teams?

The split between validation/dispatch code and implementation code. Even if you are not writing CUDA, this is a strong pattern.

### Where would this fail for many teams?

Hardware availability, operator expertise, and integration friction. A library tuned around Hopper/Blackwell-class assumptions is great if that is your fleet and pretty useless if it is not.

### Is the codebase hypey?

Not much. The README has performance claims, but the repo itself mostly shows its work.

## What is smart

- keeping the pybind entrypoint tiny and subsystem-oriented
- making the JIT cache robust enough for multi-process or distributed environments
- separating heuristics from implementation bodies
- treating tests like executable architecture examples
- building a real fused Mega MoE path instead of pretending GEMM alone is the workload
- keeping legacy Triton code present but clearly secondary

## What is flawed or weak

- the hardware target is narrow, so the audience is automatically small
- some of the API ergonomics assume you already understand scale-factor layouts and tensor memory formats deeply
- the repo is clean, but still not beginner-friendly if you are new to GPU kernel work
- Python remains mostly a shell, so debugging often pulls you quickly into C++/CUDA internals

## What we can learn / steal

- a thin public API over a deeply optimized internal engine is often the right abstraction boundary
- runtime codegen is much more maintainable when cache semantics are treated seriously
- use tests to document system composition, not just correctness
- isolate shape/layout validation at the boundary instead of scattering it across low-level implementations

## How we could apply it

If we were building our own performance-sensitive infra library, I would copy these moves:

- keep a small Python or RPC surface and push heavy specialization into a lower layer
- make generated artifacts content-addressed and atomically published
- separate dispatch heuristics from implementation bodies
- expose a few powerful tuning knobs instead of many weak ones
- write one “system test” per major subsystem that doubles as a tutorial

## Bottom line

DeepGEMM is a very good repo teardown candidate because it is both hot and substantial. The core lesson is not just “FP8 kernels go brrr.” The deeper lesson is that they designed a compact, layered kernel library where validation, dispatch, JIT compilation, and architecture-specific code each have a clean place to live.

That makes it worth studying even if you never touch these exact kernels.
