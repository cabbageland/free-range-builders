# PostHog

- Repo: `PostHog/posthog`
- URL: https://github.com/PostHog/posthog
- Date: 2026-04-25
- Repo snapshot studied: `master` @ `8911d55d2466cd5e11031f44be3cef9f650a7f9f`
- Why picked today: It is genuinely hot, broad, and AI-adjacent without being a flimsy wrapper. More importantly, the repo is worth studying because it shows a real company trying to split a giant product platform into cleaner product slices without pretending the old monolith disappeared.

## Executive summary

PostHog is not one product. It is a full product-instrumentation and product-ops stack, analytics, replay, flags, experiments, CDP, warehouse, error tracking, LLM analytics, messaging, and more, all living inside one large monorepo. The repo’s most interesting idea is not any single feature. It is the architectural migration path: keep the old Django monolith running in [`posthog/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog), keep enterprise overlays in [`ee/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/ee), but steadily move product surfaces into isolated vertical slices under [`products/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products) with explicit contracts, facades, and selective testing.

The key insight is that this repo is less “clean modular system” than “active monolith decomposition project with strong internal discipline.” That is exactly why it is valuable. Most teams have to evolve a mess, not start from a blank sheet.

The weakness is obvious too: there is still a lot of system in the old core, the API surface is enormous, and several architectural eras coexist at once. It is powerful, but the cognitive load is real.

## What they built

They built an open source product platform with many user-facing products and a lot of backend infrastructure in one repo.

The big structural zones are:

- [`posthog/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog), the legacy Django monolith, still carrying models, API routing, query execution, plugins, ClickHouse integration, task logic, settings, and a huge amount of product behavior
- [`products/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products), newer product-specific vertical slices such as feature flags, experiments, LLM analytics, dashboards, replay, workflows, and PostHog AI
- [`frontend/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/frontend), the main React app shell and shared frontend infrastructure
- [`ee/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/ee), enterprise-only overlays and extensions
- [`common/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/common), shared code and parser/runtime pieces like HogQL support and developer tooling
- [`nodejs/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/nodejs), separate Node-side runtime pieces
- [`livestream/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/livestream), a separate real-time subsystem
- [`docker/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/docker) plus the top-level compose files, local and hobby deployment machinery

This is a full-stack operating repo, not a demo app.

## Why it matters

A lot of architecture writing is about greenfield ideals. PostHog is useful because it shows what a serious product company actually does after success makes the codebase heavy:

- keep shipping
- carve product boundaries inside a monolith
- add tooling that enforces those boundaries
- separate product packages enough to unlock selective testing and ownership
- tolerate transitional ugliness instead of pretending it is not there

That makes this repo more educational than a smaller “perfectly modular” codebase.

## Repo shape at a glance

The most important structural fact is that PostHog has multiple architectural generations living side by side.

- [`posthog/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog) is the old center of gravity, Django models, APIs, query engine, tasks, settings, plugins, warehouse pieces, session recordings, feature-flag logic, and more
- [`products/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products) is the new intended product boundary system, where each product can own backend and frontend code together
- [`frontend/src`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/frontend/src) is still the shared frontend shell that stitches many products together
- [`ee/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/ee) adds enterprise behavior on top of the same system
- [`docs/internal/monorepo-layout.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/docs/internal/monorepo-layout.md) documents the target shape: legacy monolith, products, services, common, and platform
- [`products/README.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/README.md) defines the desired product package structure
- [`products/architecture.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/architecture.md) explains the rationale for facades, frozen contracts, and isolation rules

The repo is therefore best read as:

1. legacy core that still runs a lot of the company
2. product-slice migration framework
3. frontend and infra layers wrapped around both

## Layered architecture dissection

### High-level system shape

At a high level, the system looks like this:

1. the web app boots from Django settings in [`posthog/settings/web.py`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/settings/web.py)
2. core and product apps are registered together through `INSTALLED_APPS` and `PRODUCTS_APPS` in that same file
3. the giant API surface is assembled in [`posthog/api/__init__.py`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/api/__init__.py), which imports both old-core endpoints and many product endpoints
4. product-specific code increasingly lives under [`products/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products), usually with backend and frontend in the same package
5. shared query and expression infrastructure still lives in old-core subsystems like [`posthog/hogql`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/hogql)
6. deployments and local orchestration are handled through the many top-level Docker and dev-environment files such as [`docker-compose.dev.yml`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/docker-compose.dev.yml), [`docker-compose.hobby.yml`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/docker-compose.hobby.yml), and [`devenv/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/devenv)

This is not microservices-first. It is monorepo-first, monolith-plus-slices.

### Main layers

#### 1) Runtime shell and platform composition

[`posthog/settings/web.py`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/settings/web.py) is one of the most revealing files in the repo. It shows the actual runtime composition: old Django app, many new product Django apps, middleware, auth, and enterprise hooks all loaded into one web process.

That one file tells you the real truth of the architecture: product isolation is being built inside a still-shared runtime, not as separate deployables by default.

#### 2) API aggregation layer

[`posthog/api/__init__.py`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/api/__init__.py) is the central integration seam. It imports a huge list of endpoint modules from both the legacy core and newer product packages, then mounts them on shared routers.

This file is architecturally important because it shows the transitional compromise. Product code can be isolated in its own package, but the outward API plane is still centrally assembled.

#### 3) Product-vertical layer

[`products/README.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/README.md) and [`products/architecture.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/architecture.md) define the desired product shape: each product owns a Django app under `backend/`, React code under `frontend/`, a stable contract surface in `backend/facade/`, and isolated tests.

A concrete example is [`products/llm_analytics`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/llm_analytics), which carries its own backend APIs, models, queries, providers, DAGs, docs, and frontend scenes. Another is [`products/posthog_ai`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/posthog_ai), which bundles backend code, frontend code, DAGs, scripts, and even internal skills.

That is a good vertical-slice design. Ownership is visible in the filesystem.

#### 4) Legacy domain and query engine layer

Even with the product migration, a lot of the hard platform logic still lives in [`posthog/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog). The most interesting subsystem there is [`posthog/hogql`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/hogql), which contains ASTs, parser code, compiler pieces, database abstractions, printer layers, functions, and grammar files.

That matters because PostHog is fundamentally a query-heavy product. The analytics UI is only useful if the repo has a real internal query language and execution model beneath it.

#### 5) Frontend shell and package federation layer

[`frontend/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/frontend) remains the large shared React shell, while product packages contribute scenes and manifests from inside [`products/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products). The repo-level [`package.json`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/package.json) plus [`pnpm-workspace.yaml`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/pnpm-workspace.yaml) and [`turbo.json`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/turbo.json) show that the frontend and product packages are managed as a serious JS workspace, not a sidecar.

#### 6) Ops and local-environment layer

The number of deployment and environment files matters here. [`docker-compose.dev.yml`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/docker-compose.dev.yml), [`docker-compose.dev-full.yml`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/docker-compose.dev-full.yml), [`docker-compose.hobby.yml`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/docker-compose.hobby.yml), and [`docker/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/docker) tell you this repo is designed to be operated, not just read.

### Request / data / control flow

A simplified feature request path looks like this:

1. Django loads core and product apps from [`posthog/settings/web.py`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/settings/web.py)
2. routes are assembled in [`posthog/api/__init__.py`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/api/__init__.py)
3. requests land either in old-core endpoints or product-specific backend presentation/API modules
4. newer products are supposed to pass through facade boundaries described in [`products/architecture.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/architecture.md)
5. analytics-heavy features eventually rely on query infrastructure and execution logic in places like [`posthog/hogql`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/hogql), [`posthog/queries`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/queries), and ClickHouse-related code in [`posthog/clickhouse`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/clickhouse)
6. frontend scenes in [`frontend/src`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/frontend/src) and product frontend folders render the resulting product surface

The important thing is not purity. It is directional movement. The repo is trying to push more behavior into product slices without shutting down the existing platform.

## Practical questions and answers

### What is the smartest architectural move here?

Not the presence of `products/` by itself. The smart move is pairing that folder move with enforcement and testing ideas, facades, frozen dataclasses, Turbo package boundaries, and tach import rules. Lots of teams create folders. Fewer create migration mechanics.

### What is the repo’s real center of gravity today?

Still the legacy core plus shared runtime composition, especially [`posthog/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog), [`posthog/settings/web.py`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/settings/web.py), and [`posthog/api/__init__.py`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/api/__init__.py). The product-slice system is the future, but the monolith is still the present.

### What feels most durable here?

The architectural direction, not every current boundary. The repo is clearly building a sustainable product-isolation model that can survive continued growth.

### Where is the likely pain?

Cross-era complexity. The hardest thing will be understanding when behavior lives in old-core, in a new product package, or in enterprise overlays. That ambiguity is the tax of incremental migration.

### What would I copy?

The way they document the desired future state inside the repo, then wire tooling around it. [`products/architecture.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/architecture.md) is not fluff. It is a migration contract.

### What would I avoid?

Letting central API composition remain giant forever. [`posthog/api/__init__.py`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/api/__init__.py) is understandable as a transition layer, but it is also an obvious long-term choke point.

## What is smart

- treating modularization as an operational migration, not a rewrite fantasy
- documenting the target monorepo shape in [`docs/internal/monorepo-layout.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/docs/internal/monorepo-layout.md)
- defining product package conventions in [`products/README.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/README.md)
- tying architecture to testing and import enforcement in [`products/architecture.md`](https://github.com/PostHog/posthog/blob/8911d55d2466cd5e11031f44be3cef9f650a7f9f/products/architecture.md)
- keeping newer product areas as vertical slices rather than scattering files across the repo
- having a real internal query-language subsystem in [`posthog/hogql`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/posthog/hogql)
- supporting both serious cloud-scale development and hobby self-hosting from the same repo

## What is flawed or weak

- several architectural generations coexist, which increases onboarding cost
- the central API assembly layer is very large and likely hard to reason about safely
- the old-core package still appears to carry a lot of invisible weight, so the boundary story is not yet fully reflected in runtime shape
- enterprise overlays in [`ee/`](https://github.com/PostHog/posthog/tree/8911d55d2466cd5e11031f44be3cef9f650a7f9f/ee) add another axis of complexity on top of product boundaries
- the repo is so broad that “find the real implementation” is still a nontrivial task

None of this means the architecture is bad. It means the repo is real.

## What we can learn / steal

- write the migration architecture down inside the repo
- make boundaries enforceable through package and import tooling, not just conventions
- move toward vertical slices that contain backend, frontend, tests, and product docs together
- accept transitional mess, but make the intended direction unmistakable
- build a real query substrate if analytics is core to the product, not just ad hoc SQL strings everywhere

## How we could apply it

If we were evolving a growing monolith, the most reusable PostHog pattern would be:

1. define the target product boundary explicitly
2. create one filesystem unit per product
3. require a narrow public interface for that unit
4. connect testing and dependency tooling to that boundary
5. migrate incrementally while the old system still ships

That is much more believable than “just break it into services.”

## Bottom line

PostHog is worth studying because it is not pretending to be tidier than it is. The repo shows a successful product company turning a giant shared codebase into a more isolated product platform in place. The best lesson is not any one feature. It is the migration playbook: vertical slices, explicit contracts, boundary tooling, and honest coexistence with legacy core while the platform keeps moving.
