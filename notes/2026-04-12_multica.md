The interesting part is that it tries to make coding agents first-class coworkers by splitting the system into product surface, orchestration server, and machine-local runtime execution.

## Executive summary

Multica is a managed-agents workboard: assign an issue to an agent, let a machine-local daemon claim and execute the task with Claude/Codex/OpenClaw/OpenCode/Hermes, and stream status back into a shared board UI.

The repo is structurally stronger than the pitch. This is not a one-process toy. It is a fairly clear four-piece system:

1. a web UI in Next.js
2. an optional desktop shell in Electron
3. a Go backend with REST + WebSocket + task lifecycle handling
4. a local daemon that actually runs agent CLIs on user machines

The key architectural move is the daemon boundary. Multica does **not** try to run every agent inside the web app/backend process. It treats local runtimes as separate workers that register themselves, heartbeat, claim tasks, execute in isolated work dirs, and report progress/usage back.

That separation is the main reason the project feels more serious than the usual “kanban for agents” hype.

## What they built

They built an issue/project management system where agents appear as assignable teammates.

Concretely, the product combines:

- issue tracking and comments
- agent definitions and skill assignment
- runtime registration and health/usage monitoring
- real-time board updates via WebSocket
- local task execution through a daemon that shells out to coding-agent CLIs

The frontend and backend are product software. The daemon is the actual bridge to reality.

## Why it matters

Most agent products collapse under one of two bad instincts:

- treat the agent as a chat bubble with a ticket veneer
- over-centralize execution in the cloud and ignore the fact that real coding tools live on developers’ machines

Multica’s design matters because it takes the execution boundary seriously. The daemon is a proper runtime process with:

- workspace watching
- runtime registration
- heartbeat/usage reporting
- task claiming
- isolated execution environments
- per-provider adapters for different coding-agent CLIs

That is the difference between “AI PM aesthetic” and “maybe actually operable.”

## Layered architecture dissection

### High-level system shape

The core shape is:

- web/desktop client presents issues, comments, skills, runtimes, agents
- Go backend owns persistence, auth, task lifecycle, and realtime fanout
- local daemon owns actual agent execution
- external agent CLIs do the real coding work

The README’s architecture diagram is materially accurate, not pure marketing: [`README.md`](https://github.com/multica-ai/multica/blob/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/README.md).

#### 3. Backend HTTP/API + auth + workspace boundary layer

The backend route surface is registered in [`server/cmd/server/router.go`](https://github.com/multica-ai/multica/blob/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/server/cmd/server/router.go).

That file shows three important boundaries:

- public auth endpoints
- daemon API endpoints under `/api/daemon`
- protected user/workspace APIs for issues/projects/comments/runtimes/skills/etc.

This is one of the repo’s clearest strengths. The daemon routes are not bolted awkwardly onto user routes; they are treated as a separate trust domain with dedicated middleware.

This is the repo’s most important subsystem.

Key files:

- daemon core: [`server/internal/daemon/daemon.go`](https://github.com/multica-ai/multica/blob/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/server/internal/daemon/daemon.go)
- daemon HTTP client: [`server/internal/daemon/client.go`](https://github.com/multica-ai/multica/blob/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/server/internal/daemon/client.go)

The daemon:

- loads auth from CLI config
- registers watched workspaces and runtimes
- syncs workspace repos to local cache
- heartbeats periodically
- polls for tasks
- launches provider-specific agent backends
- reports progress, usage, messages, and completion/failure back to the server

That is the real center of gravity of the product.

The agent abstraction sits under [`server/pkg/agent/`](https://github.com/multica-ai/multica/tree/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/server/pkg/agent).

- Claude stream-json style
- Codex app-server / JSON-RPC
- Hermes ACP

- OpenCode event stream

That adapter pragmatism is good taste.

### Request / data / control flow

A likely happy-path flow looks like this:

1. User creates/assigns an issue in the UI.
2. Frontend talks to backend issue/task routes.
3. Backend persists state and emits workspace realtime events.
4. A watched local daemon heartbeats and polls the daemon API.
5. The daemon claims a task via [`server/internal/daemon/client.go`](https://github.com/multica-ai/multica/blob/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/server/internal/daemon/client.go).
6. The daemon selects the correct provider runtime and launches the corresponding adapter from [`server/pkg/agent/`](https://github.com/multica-ai/multica/tree/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/server/pkg/agent).
7. Progress/messages/usage are posted back to `/api/daemon/...` endpoints registered in [`server/cmd/server/router.go`](https://github.com/multica-ai/multica/blob/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/server/cmd/server/router.go).
8. Backend fans updates out through the workspace WebSocket hub in [`server/internal/realtime/hub.go`](https://github.com/multica-ai/multica/blob/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/server/internal/realtime/hub.go).
9. Frontend patches local caches through logic like [`packages/core/issues/ws-updaters.ts`](https://github.com/multica-ai/multica/blob/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/packages/core/issues/ws-updaters.ts).

This is a clean enough loop that you can picture it operating in production.

## Practical questions and answers

### How real is the execution model?

Pretty real. The daemon is not fake ceremony. It owns registration, heartbeats, task claim, execution, and reporting. That is operational substance.

### Is the frontend architecture disciplined?

Yes, reasonably. Thin app routes, shared logic in `packages/core`, reusable views in `packages/views`, and platform-specific apps as shells is the right move.

### Where would this likely hurt in production?

Three places jump out:

1. single-process WebSocket hub fanout
2. complexity of supporting many third-party agent CLIs with shifting protocols
3. eventual pain around task isolation, filesystem state, and repo cache correctness under concurrency

### Is the “skills compound over time” idea actually backed by code?

Partly. There is genuine skill CRUD and file support. Whether it truly compounds in practice depends on retrieval/versioning/composability quality, which is harder to verify from a quick teardown.

### Is this more platform or product?

It is trying to be both. That is powerful, but dangerous. The daemon/runtime layer is platform software. The board/comments/agents/skills UX is product software. Doing both well is hard.

### What is the deepest useful insight here?

The useful idea is **agent work should be modeled as a distributed systems problem, not a prompt UX problem**. Multica is strongest where it acts like orchestration software instead of chatbot furniture.

## What is smart

### 1. The daemon boundary

This is the best idea in the repo. Running agents where the tools and repos already live is much saner than forcing everything into a centralized cloud executor.

### 2. Adapter realism over false standardization

Supporting Codex, Claude, OpenClaw, OpenCode, and Hermes with separate adapters is ugly in the right way. It reflects reality.

### 3. Shared package layering

The `apps` / `packages/core` / `packages/views` / `packages/ui` split is solid. It reduces framework sprawl and makes the desktop/web dual-surface approach plausible.

### 4. Workspace-scoped realtime

The workspace-room fanout model in [`server/internal/realtime/hub.go`](https://github.com/multica-ai/multica/blob/1ee4e0501a955ce41dfd6e3e1f2c8cc62b59d0fa/server/internal/realtime/hub.go) is boring and appropriate. For this class of app, boring is good.

### 5. Self-hosting story is first-class

The project clearly wants to be used outside a pure SaaS funnel. The docs and compose setup are not hidden afterthoughts.

## What is flawed or weak

### 1. The tagline overshoots the current moat

“Your next 10 hires won’t be human” is classic startup chest-thumping. The actual interesting thing is the runtime orchestration boundary. The slogan is noisier than the engineering.

### 2. Realtime backend looks single-node minded

The in-memory hub is simple, but it suggests a future scaling seam. If the product grows, they will need a distributed event layer or sticky-session constraints.

### 3. Multi-agent provider maintenance burden is brutal

Every external CLI protocol is a moving target. This architecture buys flexibility, but also signs up for ongoing adapter breakage.

### 4. The repo is broad enough to risk dilution

Board UX, auth, comments, attachments, skills, runtimes, usage tracking, daemon, desktop app, docs site: that is a lot. It is easy for a repo like this to become “many half-good systems.”

### 5. Some of the hard problems are probably merely postponed

Questions that likely get gnarly later:

- reproducibility of task execution
- sandbox guarantees per provider
- repo mutation conflicts across concurrent tasks
- skill retrieval quality
- secret handling across local runtimes
- observability when tasks fail in subtle provider-specific ways

None of that means the design is bad. It just means the repo has entered the territory where the hard parts are genuinely hard.

## What we can learn / steal

### 1. Treat agent execution as a worker-runtime problem

If building agentic systems, steal this framing. Separate:

- orchestration/control plane
- user-facing product layer
- local execution workers
- provider-specific adapters

That is much sturdier than shoving everything into one API server and praying.

### 2. Put the framework shell on a diet

Thin route files and thick shared packages are a good pattern for multi-surface apps.

### 3. Normalize provider weirdness behind adapters

Do not force all providers into one fake universal protocol too early. Embrace adapters, then standardize internal events/results.

### 4. Use simple realtime, but only where it stays simple

Workspace-scoped room fanout is enough until it is not. Start here, but know the next step before pain hits.

### 5. Skills should be file-backed, not just prompt-backed

This repo’s skill model suggests a better default: reusable operational knowledge often wants multiple files/config artifacts, not one magic string.

## How we could apply it

For our own systems, the reusable pattern is:

- a central planning/product surface where tasks live
- a small backend that owns state, auth, and lifecycle transitions
- local machine agents as workers with heartbeat + claim + usage report contracts
- provider adapters as replaceable modules
- realtime only for user-visible coordination state, not for everything

If building an agent team product, I would copy the control-plane / local-daemon split almost immediately.

I would **not** copy the hype posture. I would sell the boring power: local execution, explicit runtime health, reproducible task lifecycle, and provider-agnostic orchestration.

## Bottom line

Multica is worth studying because it understands the central truth that a lot of agent products dodge: the hard part is not “chatting with an agent,” it is coordinating execution across machines, repos, tools, and humans without turning the whole thing into mush.

The repo’s strongest move is the daemon-centric runtime architecture. Its weakest risk is trying to be a full product suite and a runtime platform at the same time.

Still: this is a real repo teardown candidate, not a fake-interesting wrapper. The main thing to steal is the boundary design.
