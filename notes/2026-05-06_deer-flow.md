# DeerFlow

- Repo snapshot studied: `main@1336872b15c25d45ebcb7c1cf72369c2bdd53187`
- Why picked today: It is still hot, clearly AI-related, and more structurally interesting than the average “agent framework” repo because it is trying to be a full harness product, not just a prompt loop with tools.

## Executive summary
DeerFlow is an agent harness that wraps a LangGraph runtime, a separate FastAPI gateway, a Next.js workspace UI, configurable model/tool providers, MCP integration, persistent memory, sandbox execution, and subagent delegation into one installable product.

The key thing to understand is that the repo is organized like an agent operating environment, not a single chatbot. The backend is doing real runtime plumbing: per-thread workspaces, middleware-ordered execution, tool exposure control, async subagent execution, upload/artifact handling, and multi-surface delivery through web UI plus IM channels.

The strongest idea here is not “agents with tools”. It is the explicit runtime shell around them. The main weakness is that this much flexibility creates a wide security and maintenance surface. DeerFlow knows that, but the architecture still looks like a system that will keep paying complexity tax.

## What they built
They built a full-stack open-source agent product with three visible layers:

- a LangGraph-driven lead-agent runtime defined through [`backend/langgraph.json`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/langgraph.json) and assembled in [`backend/packages/harness/deerflow/agents/factory.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/agents/factory.py)

- a Next.js web workspace rooted at [`frontend/src/app/workspace/page.tsx`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/frontend/src/app/workspace/page.tsx) and wrapped by [`frontend/src/app/workspace/workspace-content.tsx`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/frontend/src/app/workspace/workspace-content.tsx)

Under that product shell, they also built:

## Why it matters
A lot of agent repos are still demos with a nice landing page. DeerFlow is trying to solve the less glamorous parts that determine whether an agent product is actually usable: thread state, tools, uploads, artifacts, environment isolation, model configuration, UI, and channel delivery.

That makes it worth studying because the repo exposes what it takes to turn “an agent” into “a system people can operate”.

## Layered architecture dissection
### High-level system shape
The clean mental model is:

1. config and bootstrapping
2. lead-agent runtime assembly
3. middleware-governed execution around state, sandbox, and memory
4. tool, MCP, skill, and subagent extension plane
5. delivery surfaces through gateway API, web UI, and IM channels

The architecture note in [`backend/docs/ARCHITECTURE.md`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/docs/ARCHITECTURE.md) basically confirms that model.

### Main layers
**1. Product bootstrap and operator setup**

- [`README.md`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/README.md) sells the system as a super-agent harness.
- [`Makefile`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/Makefile) is the operator-facing control panel: `setup`, `doctor`, `dev`, Docker commands, config upgrade, sandbox prep.
- [`config.example.yaml`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/config.example.yaml) is the main control surface and shows how much of the product is policy-driven.

**2. Agent assembly and runtime core**

- [`backend/langgraph.json`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/langgraph.json) binds the lead agent and checkpointer into LangGraph.
- [`backend/packages/harness/deerflow/agents/factory.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/agents/factory.py) is the core assembly seam. It turns features into an ordered middleware chain and a tool bundle.
- The interesting part is that the repo treats middleware order as architecture, not implementation detail.

**3. Execution envelope around the model**

- The middleware chain described in [`backend/README.md`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/README.md) and [`backend/docs/ARCHITECTURE.md`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/docs/ARCHITECTURE.md) handles thread directories, uploads, sandbox acquisition, summarization, todos, title generation, memory, image viewing, and clarification interception.
- That means “agent behavior” is really a stack of runtime services wrapped around the model call, not just prompt plus tools.

**4. Tool and extension plane**

- [`backend/packages/harness/deerflow/tools/tools.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/tools/tools.py) resolves configured tools, suppresses unsafe host-bash exposure by default, injects built-ins, adds MCP tools, and optionally exposes ACP agent invocation.
- [`backend/packages/harness/deerflow/mcp/`](https://github.com/bytedance/deer-flow/tree/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/mcp) is the MCP layer.
- [`backend/packages/harness/deerflow/community/`](https://github.com/bytedance/deer-flow/tree/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/community) contains adapters for search, scraping, image search, sandboxes, and provider-specific integrations.

**5. Subagent and concurrency layer**

- [`backend/packages/harness/deerflow/subagents/executor.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/subagents/executor.py) is real orchestration code, not a placeholder. It manages statuses, task ids, thread pools, an isolated persistent event loop, tool filtering, timeouts, and background execution state.
- This is one of the clearest signs the project is optimizing for multi-step, potentially long-lived work.

**6. Gateway and product API layer**

- [`backend/app/gateway/app.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/app/gateway/app.py) wires app startup, config loading, auth middleware, runtime initialization, channel service startup, and orphan-thread migration.
- The router set under [`backend/app/gateway/routers/`](https://github.com/bytedance/deer-flow/tree/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/app/gateway/routers) exposes models, memory, uploads, runs, threads, skills, suggestions, channels, and more.

**7. Frontend application layer**

- [`frontend/src/app/workspace/`](https://github.com/bytedance/deer-flow/tree/1336872b15c25d45ebcb7c1cf72369c2bdd53187/frontend/src/app/workspace) is the main workspace shell.
- [`frontend/src/core/`](https://github.com/bytedance/deer-flow/tree/1336872b15c25d45ebcb7c1cf72369c2bdd53187/frontend/src/core) is where the frontend’s real structure lives: agents, API, artifacts, auth, memory, models, skills, threads, todos, uploads, settings.
- [`frontend/src/core/agents/api.ts`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/frontend/src/core/agents/api.ts) shows the frontend is built against a serious backend surface, not a toy single-endpoint chat API.

### Request / data / control flow
A typical web run appears to work like this:

1. the user interacts with the Next.js workspace under [`frontend/src/app/workspace/`](https://github.com/bytedance/deer-flow/tree/1336872b15c25d45ebcb7c1cf72369c2bdd53187/frontend/src/app/workspace)
2. frontend API helpers in [`frontend/src/core/`](https://github.com/bytedance/deer-flow/tree/1336872b15c25d45ebcb7c1cf72369c2bdd53187/frontend/src/core) call the backend routes
3. Nginx, as described in [`backend/README.md`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/README.md), splits traffic between frontend, Gateway API, and LangGraph server
4. the LangGraph side creates the lead agent via [`backend/packages/harness/deerflow/agents/factory.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/agents/factory.py)
5. middleware initializes thread-local storage, uploads, sandbox, memory context, and optional plan/title/image services
6. tool loading in [`backend/packages/harness/deerflow/tools/tools.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/tools/tools.py) determines the execution surface the model can actually reach
7. if a task is delegated, the background machinery in [`backend/packages/harness/deerflow/subagents/executor.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/subagents/executor.py) schedules and tracks the child run
8. generated artifacts, uploads, and thread metadata are then served back through gateway routes

That is a pretty full control loop.

## Important components
**The middleware chain is the product brain**

The most reusable idea in DeerFlow is that runtime concerns are split into ordered middleware instead of piling into one giant turn loop. Thread directories, uploads, sandbox acquisition, summarization, todos, title generation, memory, and clarification are all separate runtime concerns. That is cleaner than the normal “everything in agent.py” pattern.

**Tool exposure is treated as policy**

[`backend/packages/harness/deerflow/tools/tools.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/tools/tools.py) is more important than it first looks. It prevents unsafe default bash exposure, deduplicates tool names, conditionally adds vision tools, folds in MCP tools, and can surface ACP agents. That file is basically the runtime capability firewall.

**Subagents are implemented as real background work**

[`backend/packages/harness/deerflow/subagents/executor.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/subagents/executor.py) uses thread pools, async event loops, status tracking, and cancellation state. That matters because many repos claim “subagents” when they really mean nested synchronous calls.

**The gateway is doing more than API passthrough**

[`backend/app/gateway/app.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/app/gateway/app.py) handles lifecycle, runtime dependency setup, auth, admin bootstrapping, migration of orphaned thread ownership, and IM service startup. It is an operational control plane, not just an HTTP wrapper.

**The frontend is organized like a product client**

The `frontend/src/core/*` split into domains like agents, memory, skills, models, uploads, and threads suggests a real UI contract, not a single chat page with incidental helpers.

## Practical questions and answers
**Is this mostly a LangGraph wrapper?**

No. LangGraph is a core dependency, but DeerFlow’s real work is in the runtime shell around it.

**What is the best engineering move in the repo?**

Treating thread-local workspace, tool exposure, memory, and subagents as explicit runtime systems instead of magical prompt features.

**What is the most important thing to copy?**

The ordered middleware model plus the capability-aware tool loader.

**Where will this hurt?**

Security review, config sprawl, and product consistency. Systems that expose bash, uploads, MCP, subagents, channels, and web UI at once are powerful but easy to get subtly wrong.

**Does the repo feel production-minded?**

More than most trending agent repos, yes. It has obvious scars around auth, shutdown, thread ownership, and deployment. But it still feels like a fast-moving platform rather than a settled one.

## What is smart
- The backend is architected as a runtime envelope around the model, not just a model caller.
- The middleware ordering in [`backend/packages/harness/deerflow/agents/factory.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/agents/factory.py) is a good pattern for keeping agent infrastructure legible.
- The tool-loading logic in [`backend/packages/harness/deerflow/tools/tools.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/tools/tools.py) is careful in the right places.
- The subagent executor is implemented like infrastructure, not hype.
- The gateway owns real operational concerns, including channel startup and auth-state migration.
- The repo offers multiple surfaces, but they still point back to one runtime model.

## What is flawed or weak
- The product surface is extremely broad for one open-source repo: UI, API, LangGraph runtime, skills, MCP, auth, channels, sandboxes, memory, subagents, deployment. That breadth is impressive, but expensive.
- Security posture is necessarily delicate because the whole point is to expose powerful execution paths.
- The configuration surface is flexible but can easily become a support burden.
- The architecture has clean concepts, but many of them still live in one large backend package, so long-term maintainability will depend on continued boundary discipline.
- Like many agent products, a lot of reliability still depends on the model making decent decisions inside a carefully engineered cage.

## What we can learn / steal
- Use middleware order as a first-class architecture tool for agent systems.
- Separate product API concerns from core agent harness logic.
- Treat tool exposure as policy that is computed, filtered, and audited.
- Build subagent execution as actual background infrastructure with status and cancellation.
- Give agent threads explicit per-thread filesystems and artifact paths.

## How we could apply it
If we were building our own agent runtime, I would copy these ideas first:

1. the feature-to-middleware assembly pattern in [`backend/packages/harness/deerflow/agents/factory.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/agents/factory.py)
2. the capability-gated tool loading in [`backend/packages/harness/deerflow/tools/tools.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/tools/tools.py)
3. the real background subagent executor in [`backend/packages/harness/deerflow/subagents/executor.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/packages/harness/deerflow/subagents/executor.py)
4. the split between runtime core and gateway product API shown by [`backend/app/gateway/app.py`](https://github.com/bytedance/deer-flow/blob/1336872b15c25d45ebcb7c1cf72369c2bdd53187/backend/app/gateway/app.py)

I would be more cautious about copying the full product breadth unless we were ready to own the security and support load that comes with it.

## Bottom line
DeerFlow is interesting because it treats agents as an operating system problem.

The key insight is that the repo’s real value is the execution envelope around the model: middleware, tool policy, per-thread isolation, and background subagents. That is the part worth studying and stealing.
