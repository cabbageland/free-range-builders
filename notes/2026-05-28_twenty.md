# Twenty

- Date: 2026-05-28

- Why picked today: It is one of GitHub’s hottest repos today, but more importantly it is a serious product repo, not an AI-wrapper spike. It is a large open-source CRM with a real application platform, a big monorepo, and enough architectural weight to reward close study.

## Executive summary
Twenty is not just “open-source Salesforce.” It is trying to be a CRM platform that technical teams can extend and version like application code. The repo reflects that ambition: there is a product frontend, a large NestJS backend, a metadata layer for workspace customization, a UI kit, an SDK, a scaffolder, and a full self-hosting surface.

The strongest idea in the repo is that CRM customization is treated as product infrastructure, not as a pile of ad hoc admin settings. The risk is the obvious one: once your CRM becomes a platform, every layer gets heavier, and this repo is already carrying a lot of weight.

## What they built
They built a monorepo for a customizable CRM platform with:
- a React frontend in [`packages/twenty-front/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-front),
- a NestJS backend in [`packages/twenty-server/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server),
- a reusable UI system in [`packages/twenty-ui/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-ui),
- shared types and workflow primitives in [`packages/twenty-shared/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-shared),
- app/SDK tooling in [`packages/twenty-sdk/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-sdk), [`packages/twenty-cli/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-cli), and [`packages/create-twenty-app/`](https://github.com/twentyhq/twenty/tree/main/packages/create-twenty-app),
- and deployment assets in [`packages/twenty-docker/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-docker).

README framing matters here: the product pitch is not only CRM records and pipelines, it is “build your CRM as code.” The scaffolding example in [`README.md`](https://github.com/twentyhq/twenty/blob/main/README.md) and the app tooling packages make that a structural repo fact, not just marketing copy.

## Why it matters
Most CRMs become either rigid SaaS software or sprawling enterprise customization traps. Twenty is taking a more developer-native path:
- the CRM itself is open source,
- the workspace can be extended as code,
- the monorepo includes both product runtime and extension tooling,
- and self-hosting is treated as a first-class path.

That makes it interesting beyond CRM. It is really a study in how to turn a line-of-business app into a programmable platform.

## Layered architecture dissection
### High-level system shape
There are five main systems here:
1. a backend application engine,
2. a product frontend,
3. a metadata and extension platform,
4. a shared component/type/tooling layer,
5. and a self-host/deployment layer.

### Main layers
**1. Monorepo orchestration layer**
- [`package.json`](https://github.com/twentyhq/twenty/blob/main/package.json) defines the Yarn workspace packages and the multi-process local start command.
- [`nx.json`](https://github.com/twentyhq/twenty/blob/main/nx.json) is doing real organizational work: build/test/lint defaults, cache boundaries, storybook/test exclusions, and project-wide task conventions.

This matters because the repo is too large to manage casually. Nx is part of the architecture, not just tooling garnish.

**2. Backend engine and API layer**
- [`packages/twenty-server/src/main.ts`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-server/src/main.ts) bootstraps the Nest app, session storage, upload handling, front-config generation, and environment-dependent HTTPS.
- [`packages/twenty-server/src/app.module.ts`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-server/src/app.module.ts) shows the real backend composition: GraphQL, REST, MCP, metrics, auth context middleware, workspace cache, metadata modules, and ORM wiring.
- [`packages/twenty-server/src/engine/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server/src/engine) is the backend substrate: API transport, auth, config, metrics, middleware, metadata, guards, dataloaders, search, messaging, workflow, and more.

A notable architectural split appears inside the server:
- [`src/engine/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server/src/engine) looks like reusable runtime infrastructure and platform internals.
- [`src/modules/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server/src/modules) looks like business capability modules such as calendar, messaging, connected accounts, dashboard, workflow, and workspace members.

That is a healthy split. The repo distinguishes platform machinery from product features.

**3. Frontend product layer**
- [`packages/twenty-front/src/index.tsx`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-front/src/index.tsx) is thin, which is good. It mainly mounts the app and loads shared styling.
- [`packages/twenty-front/src/modules/app/components/App.tsx`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-front/src/modules/app/components/App.tsx) shows the frontend shell: Jotai store, i18n, error boundaries, icon provider, exception handling, and router handoff.
- [`packages/twenty-front/folderStructure.json`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-front/folderStructure.json) is especially revealing. It codifies frontend module layout rules for hooks, utils, states, types, services, contexts, and components.

That file is one of the most interesting things in the repo. They are trying to make frontend structure enforceable, not merely aspirational.

**4. Platform and extension layer**
- [`packages/twenty-sdk/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-sdk) packages the app-definition and SDK surface.
- [`packages/create-twenty-app/`](https://github.com/twentyhq/twenty/tree/main/packages/create-twenty-app) turns “build on top of Twenty” into a paved path instead of a docs chore.
- [`packages/twenty-shared/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-shared) centralizes shared workflow, metadata, AI, workspace, and utility code.

This is the key platform move: the CRM product and the extension product are built in the same repo, with shared primitives underneath.

**5. Delivery and self-hosting layer**
- [`packages/twenty-docker/docker-compose.yml`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-docker/docker-compose.yml), [`packages/twenty-docker/helm/twenty/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-docker/helm/twenty), and [`packages/twenty-docker/k8s/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-docker/k8s) show that deployment is not bolted on.
- The presence of Grafana, OpenTelemetry collector config, Podman support, and Terraform-oriented infra directories suggests they expect real operator use, not just hobby docker-compose setups.

### Request / data / control flow
A simplified flow looks like this:
1. the browser boots [`packages/twenty-front/src/index.tsx`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-front/src/index.tsx) and hands control to the frontend app shell,
2. the frontend routes through app modules and shared UI/state providers,
3. requests hit the Nest backend defined by [`packages/twenty-server/src/main.ts`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-server/src/main.ts),
4. [`packages/twenty-server/src/app.module.ts`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-server/src/app.module.ts) routes those requests into GraphQL, REST, admin, metadata, or MCP surfaces,
5. engine modules provide auth, middleware, metadata, ORM, and cross-cutting services,
6. business modules under [`packages/twenty-server/src/modules/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server/src/modules) execute product-specific behavior like messaging, workflows, calendar sync, dashboards, and connected accounts.

The important architectural idea is that “CRM logic” is not a single blob. It is layered over a metadata-driven platform core.

## Important components
**The server composition root**
[`packages/twenty-server/src/app.module.ts`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-server/src/app.module.ts) is where the repo tells the truth about itself. It is not just a CRUD backend. It wires GraphQL, REST, MCP, metrics, dataloaders, metadata, cache layers, ORM modules, and static front serving.

**The engine versus modules split**
[`packages/twenty-server/src/engine/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server/src/engine) versus [`packages/twenty-server/src/modules/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server/src/modules) is the cleanest structural decision I found. Engine is platform substrate, modules are business capabilities.

**The frontend structure contract**
[`packages/twenty-front/folderStructure.json`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-front/folderStructure.json) is a real gem. It formalizes how frontend modules recurse and where hooks, utils, states, services, and validation schemas should live. That is the sort of small-but-serious repo hygiene that usually predicts better long-term maintainability.

**The shared UI system**
[`packages/twenty-ui/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-ui) is substantial enough that the frontend is not just app code; it is leaning on an internal design-system package. That usually pays off in a product this broad.

**The app platform toolchain**
[`packages/twenty-sdk/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-sdk), [`packages/twenty-cli/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-cli), and [`packages/create-twenty-app/`](https://github.com/twentyhq/twenty/tree/main/packages/create-twenty-app) are what turn Twenty from “an OSS CRM” into “a CRM platform you can build against.”

## Practical questions and answers
**Is this mostly a CRM app or a CRM platform?**
Both, but the repo leans platform. The SDK, app scaffolder, metadata-heavy backend, and extension-focused README make that clear.

**Where is the architectural center of gravity?**
The server. Specifically the split between [`src/engine/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server/src/engine) and [`src/modules/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server/src/modules).

**What is the most reusable idea here?**
Treat business-software customization as code artifacts with shared primitives, not just runtime configuration screens.

**Where would this get painful?**
Metadata complexity, upgrade safety, and cross-package coordination. Large configurable business apps accumulate edge cases fast.

## What is smart
- The repo uses monorepo boundaries that map to real product boundaries.
- Backend engine code is separated from business modules instead of mixing framework internals with domain code.
- Frontend structure is codified in [`folderStructure.json`](https://github.com/twentyhq/twenty/blob/main/packages/twenty-front/folderStructure.json), which is unusually disciplined.
- Self-hosting is treated seriously, with Compose, Helm, K8s, Podman, and observability assets.
- The extension path is paved by actual packages, not hand-wavy “API-first” claims.

## What is flawed or weak
- The repo is huge. That is not automatically bad, but it means contributor onboarding cost will be real.
- The platform ambition is high enough that conceptual sprawl is a risk: CRM app, app platform, metadata system, SDK, AI/agents, and infra all in one place.
- [`packages/twenty-server/src/engine/`](https://github.com/twentyhq/twenty/tree/main/packages/twenty-server/src/engine) is broad enough that understanding ownership boundaries inside it may be hard for new contributors.
- The frontend likely benefits from the structure rules, but it also suggests the UI surface is large enough to need governance tooling just to stay coherent.

## What we can learn / steal
- Separate platform substrate from business modules explicitly.
- Make extension tooling live beside the product, not in a separate afterthought repo.
- Encode folder and module discipline as machine-readable rules when the frontend gets large.
- Treat deployment and self-hosting as real product surfaces if you want serious adoption.
- Use monorepo tooling only when you actually have monorepo-scale coordination problems. Twenty does.

## How we could apply it
If we were building a configurable internal app platform, I would copy three things:
1. the engine-versus-feature split on the backend,
2. the codified frontend module structure,
3. the fact that scaffolding/SDK/deployment all ship as first-class packages.

I would probably simplify the first version by narrowing the product surface. Twenty can justify its breadth because the CRM itself is the platform.

## Bottom line
Twenty is worth studying because it is building the thing many teams claim to want but few repos really support: business software that is both usable out of the box and programmable like application code.

My main takeaway is simple: the interesting part is not that it is an open CRM. The interesting part is that the repo is organized around making customization an actual software discipline.
